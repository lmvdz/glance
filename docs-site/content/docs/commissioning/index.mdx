In a human org a job opens, someone is hired, and they're onboarded. omp-squad
replaces the hire with **the OS authoring its own specialized worker**. A fuzzy job
spec goes in; a builder agent writes a small, scoped, deployable [Flue](https://flueframework.com)
worker; and вҖ” only if the worker passes an **acceptance gate** вҖ” it's onboarded into the
live fleet as a `flue-service` member.

```text
author вҶ’ gate вҶ’ onboard
```

This is the principle-of-least-privilege answer to a job: instead of spawning another
full-shell omp coding agent for every ability, the OS mints a worker with a narrow toolset
and a recorded capability allowlist.

## The loop is a workflow

The `author вҶ’ gate вҶ’ onboard` sequence is not an imperative method body вҖ” it's a
[workflow graph](/docs/workflows) (`workflows/commission/workflow.fabro`) run on the same
pure engine as `plan-implement`, driven by `CommissionExecutor`
(`src/workflow/commission-executor.ts`). Its three action nodes bind to the manager's
domain hooks; `runAgent` / `runCommand` / `humanGate` throw, because the commission graph
uses action nodes only.

```text
        в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
        в”Ӯ            (Reject)           в”Ӯ
        в–ј                               в”Ӯ
JOB вҶ’ AUTH в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–ә GATE в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                          в”Ӯ
                          в”Ӯ Pass (outcome=succeeded)
                          в–ј
                         ONB вҶ’ exit
```

```text title="workflows/commission/workflow.fabro"
author   [label="Author",          action="author",   max_visits=2]
gate     [label="Acceptance gate",  action="validate", goal_gate=true, retry_target="author"]
onboard  [label="Onboard",          action="onboard"]

start -> author -> gate
gate -> onboard [label="Pass", condition="outcome=succeeded"]
gate -> author  [label="Reject"]
onboard -> exit
```

A failed gate (`outcome=failed`) skips the conditioned `Pass` edge and follows the
unconditioned `Reject` edge back to `author`, which re-authors the worker вҖ” **feeding the
prior failure forward** as guidance. `author` carries `max_visits=2`, so the engine bounds
the loop: when authoring would exceed its visit cap, nothing routes and the run fails, so
nothing is onboarded. See [gates & loops](/docs/workflows/gates) for the routing rules.

<Callout type="info">
`CommissionExecutor` keeps the latest `GateReport` on `report` and the onboarded member on
`member`. The manager's `commission()` reads `member` after the run: set вҶ’ `{ ok: true, вҖҰ }`;
absent вҶ’ the candidate was rejected and `member` is omitted.
</Callout>

## The CLI

```bash title="Deterministic worker (no model)"
omp-squad commission extract-emails \
  --purpose "Extract email addresses from payload.text; return { emails, count }." \
  --accept-payload '{"text":"a@x.io b@y.org"}' --accept-expect '{"count":2}'
```

`commission` (alias `hire`) parses the flags into a `CommissionSpec`, POSTs a
`{ type: "commission", spec }` command to the daemon, then prints one line per gate check
(`вң“` pass, `вң—` fail, `В·` skip). On success it reports the onboarded member, its id, and the
worker dir; on a failed gate it exits non-zero and leaves the worker on disk for inspection.

  }}
/>

The daemon dispatches the command to `SquadManager.commission(spec, opts, actor)`, which
resolves a worker dir (`<stateDir>/workers/<name>/` by default), builds the
`CommissionExecutor` hooks, and runs the graph. The `actor` is the authorization handle for
*who may onboard a worker*.

## Two fleet classes, one seam

| | `omp-operator` | `flue-service` |
|---|---|---|
| Runtime | `omp --mode rpc` child in a git worktree | a Flue worker invoked via `flue run` |
| Role | interactive, steerable dev sessions | scoped, bounded automation |
| Builder | spawned directly by an operator | **authored by an architect agent** |
| Driver | `RpcAgent` | `FlueServiceDriver` |

Both implement the one [`AgentDriver`](/docs/architecture/agent-driver) contract, so the
`SquadManager` derives status, buffers a transcript, and federates presence identically.
The DTO's `kind` (`"flue-service"`) is the only thing a surface needs to tell a commissioned
worker apart from an interactive operator.
