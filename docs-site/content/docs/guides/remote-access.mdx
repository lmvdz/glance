The dashboard is a PWA you can install on your phone and drive from anywhere ‚Äî and it **pushes
you a notification the moment an agent needs a human**, so you supervise by exception instead of
watching a screen. This page covers getting it off `localhost` safely.

## Bind beyond loopback

By default the daemon binds `127.0.0.1`. To reach it from another device, bind a routable
address:

```bash
glance up --no-tui --host 0.0.0.0
```

  type={{
    "--host / GLANCE_HOST": { description: "Bind address; 0.0.0.0 exposes on the LAN/tailnet. The env var wins over the flag.", type: "string", default: "127.0.0.1" },
    "--port / GLANCE_PORT": { description: "Daemon port.", type: "number", default: "7878" },
    "GLANCE_TLS_CERT + GLANCE_TLS_KEY": { description: "PEM paths ‚Äî terminate TLS in-process (both required). Switches the scheme to https.", type: "path", default: "plain HTTP" },
    "GLANCE_STATE_DIR": { description: "Where the token, VAPID keys, and subscriptions live. Default: ~/.glance (an existing legacy ~/.omp/squad is kept).", type: "path", default: "~/.glance" },
  }}
/>

On boot the daemon prints every URL it's reachable on ‚Äî loopback plus each non-internal IPv4
when bound to `0.0.0.0` ‚Äî each with a one-tap `?token=‚Ä¶` sign-in link.

## The access token

The control plane can spawn agents, land code, and re-exec itself, so **access is token-gated**
(`src/auth.ts`). On first run the daemon generates a bearer token, persists it at
`<state-dir>/access-token` (mode `0600`), and prints it on boot:

```text
glance daemon running
  dashboard: http://0.0.0.0:7878
  access token: Bdnw7‚Ä¶IHo
  open from any device on this network (tap to sign in):
    http://192.168.1.20:7878/?token=Bdnw7‚Ä¶IHo
```

Every `/api` request carries it as `Authorization: Bearer <t>`; the WebSocket carries it as a
`?token=` query param (browsers can't set headers on a WS handshake). Comparison is
constant-time. The token is **required even on loopback**. Only the static shell ‚Äî plus the
manifest, service worker, and icons needed to bootstrap the PWA ‚Äî is public; everything else is
gated. The dashboard captures `?token=` into `localStorage`, strips it from the URL, and shows a
sign-in overlay if it's missing or rejected.

Two optional pre-shared tokens complement the generated one: `GLANCE_OPERATOR_TOKEN` (write
access for scripts/CI) and `GLANCE_VIEWER_TOKEN` (read-only).

## Multi-tenant mode (`DATABASE_URL`) ‚Äî opt-in

By default the daemon runs in **file mode** ‚Äî the token-gated single-operator tool above. Set
`DATABASE_URL` (`postgres://‚Ä¶` or `sqlite:<path>`) and it boots in **DB mode**: a multi-tenant
identity layer backed by [BetterAuth](https://better-auth.com).

- **Accounts + sessions replace the bearer token.** Email + password sign-in with httpOnly,
  `SameSite=Lax` cookie sessions, rate-limited against brute force ‚Äî plus **GitHub social
  login** (`GITHUB_CLIENT_ID`/`GITHUB_CLIENT_SECRET`) and **WorkOS SSO with SCIM directory
  provisioning** (`WORKOS_CLIENT_ID`/`WORKOS_API_KEY`/`WORKOS_WEBHOOK_SECRET`).
- **Sign-up is closed by default** ‚Äî set `GLANCE_ALLOW_SIGNUP=1` for self-service registration
  (self-serve users get a personal workspace); otherwise the bootstrap admin provisions and
  invites.
- **Least privilege by default.** A signed-in user with no active org is a read-only `viewer`;
  operator/admin tiers come only from an org membership (owner/admin ‚Üí `admin`, member ‚Üí
  `operator`). Self-minting an org is disabled, so a remote user can't escalate.
- **Bootstrap (break-glass).** A request **from loopback carrying the daemon's bearer token**
  resolves to `admin` even in DB mode, so the operator on the machine can create the first org
  and invite members; off-box requests must use sessions.
- **`BETTER_AUTH_SECRET` is required** on a non-loopback DB-mode bind ‚Äî the daemon refuses to
  boot without a strong value. Behind a TLS tunnel, set `BETTER_AUTH_URL` to the external
  `https://` origin so logins pass the origin check and the session cookie is `Secure`.
- **Org management** lives in the dashboard's Settings surface: rename, invite by email,
  member roles, join policy, and WorkOS join-request approval.

<Callout type="warn">
File mode remains the default ‚Äî and the autonomous factory's native habitat: the TUI, the
standing supervisor, and the local federation bus are file-mode features. DB mode previews the
multi-tenant SaaS surface; today all org members of a DB-mode daemon still share one fleet.
</Callout>

## Install the PWA on your phone

<Callout type="warn">
**HTTPS is required** to install the PWA and receive background push ‚Äî browsers only allow
service workers and Web Push in a secure context. `http://localhost` is exempt; a LAN IP is not.
</Callout>

<Tabs items={['Tailscale (recommended)', 'In-process TLS']}>
<Tab value="Tailscale (recommended)">

Front the daemon with a real cert ‚Äî no in-process TLS needed:

```bash
glance up --no-tui            # bound to localhost is fine
tailscale serve --bg 7878        # ‚Üí https://<machine>.<tailnet>.ts.net
```

Open that URL on your phone (same tailnet), append `?token=‚Ä¶` once, **Add to Home Screen**, and
allow notifications. Tailnet ACLs gate who can reach it; the token gates the rest.

</Tab>
<Tab value="In-process TLS">

Point `GLANCE_TLS_CERT` / `GLANCE_TLS_KEY` at a cert and bind `--host 0.0.0.0`:

```bash
GLANCE_TLS_CERT=./cert.pem GLANCE_TLS_KEY=./key.pem \
  glance up --no-tui --host 0.0.0.0
```

Self-signed works but browsers warn (and some refuse a service worker on an untrusted cert), so a
real cert / tailnet is smoother.

</Tab>
</Tabs>

On the phone the unified nav collapses to a drawer, the **attention queue is the landing view**
when something's waiting, and one tap answers an approval or question.

## Web Push

When an agent transitions to a state that needs a human (`input` / `error`), the daemon pushes an
encrypted notification to every subscribed device ‚Äî the phone buzzes even with the app closed,
and tapping it deep-links to that agent (`#/agent/<id>`).

It's implemented **dependency-free** in `src/push.ts`: RFC 8291 message encryption
(`aes128gcm`) + RFC 8292 VAPID. The daemon generates and persists a VAPID keypair in the state
dir; the browser fetches the public key from `GET /api/push/key` and enrolls via
`POST /api/push/subscribe` once notifications are granted.

| Env | Meaning | Default |
|---|---|---|
| `GLANCE_PUSH_SUBJECT` | VAPID `sub` contact (`mailto:` or `https:`) | `mailto:squad@localhost` |

Server-side alerts are throttled per agent and seeded from the first roster, so a reconnect or
replay never fires a notification storm.

## Hands-free auto-supervisor

So a fleet keeps moving while you're away, the daemon runs a **standing auto-supervisor**
(`src/supervisor.ts`) ‚Äî a model-backed loop that answers blocked agents for you. It's **on by
default**: the daemon starts it unless `GLANCE_AUTO_SUPERVISE=0` or `--no-supervise` is set.
The boot line reports its state (`autonomy: session-tracker on ¬∑ auto-supervisor on`).

It opens the daemon's own WebSocket and, for every agent in `input`, decides and submits an
answer per pending request:

- **Approves** normal development actions ‚Äî edits, reads, running tests/builds, installing deps,
  committing inside the worktree, spawning subagents ‚Äî and picks the gate option that advances
  the work.
- **Denies** only clearly destructive or irreversible actions that **escape the worktree**
  (force-pushing main, deleting unrelated data, publishing a release, touching prod). The
  worktree is the blast radius; when in doubt inside it, it approves.
- Falls back to a deterministic choice if the model is unavailable or returns junk.

| Env / flag | Meaning | Default |
|---|---|---|
| `GLANCE_AUTO_SUPERVISE` | Set to `0` to disable the standing supervisor loop | on |
| `--no-supervise` | Disable it for this run | ‚Äî |
| `GLANCE_SUPERVISE_MODEL` | Model the answerer uses | omp default |

<Callout type="info">
This standing loop (`GLANCE_AUTO_SUPERVISE`, underscore, **on by default**) is distinct from
the in-manager deterministic `GLANCE_AUTOSUPERVISE` path (no underscore, **off by default**)
described in [Autonomy](/docs/architecture/autonomy). The daemon also runs a session-tracker that
surfaces raw `omp` sessions in presence.
</Callout>
