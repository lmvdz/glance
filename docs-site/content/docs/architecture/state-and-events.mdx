The manager holds live state in memory and projects two things outward: a durable
**roster config** on disk, and a **`SquadEvent`** stream to surfaces. Surfaces push
mutations back as **`ClientCommand`**s. This page is the wire reference for all three.

## Persistence

Roster config persists to `<stateDir>/state.json`. The state dir defaults to `~/.glance`
(an existing legacy `~/.omp/squad` keeps working; `GLANCE_STATE_DIR` overrides — see
[Configuration](/docs/reference/configuration#state-directory) for the exact resolution
order). It also holds the access token, per-agent run receipts, resume digests, and
commissioned-worker dirs.

```text title="default state dir"
~/.glance/
  state.json            # roster config + transcripts + features
  access-token          # bearer token (mode 0600)
  sockets/<id>.sock      # one agent-host UDS per live agent
  worktrees/             # per-agent git worktrees
```

Writes are **serialized and atomic**. `persist()` chains each write after the last one
through `writeChain` (so two writes never interleave and `await persist()` is a real
durability barrier), and `persistNow()` writes to `state.json.tmp` then renames — a
reader or a restart never sees a partial file.

```ts title="src/squad-manager.ts — persistNow()"
const agents = [...this.agents.values()].map((r) => r.options);   // PersistedAgent[]
const transcripts = …;                                            // id → TranscriptEntry[]
const features = [...this.featureStore.values()];                 // PersistedFeature[]
await fs.writeFile(tmp, JSON.stringify({ version: 1, agents, transcripts, features }, null, 2));
await fs.rename(tmp, this.stateFile);
```

<Callout type="info">
  Transcripts are redacted at the single `append()` chokepoint **before** they reach
  the in-memory buffer, `state.json`, or an emitted `transcript` event — secrets never
  hit any of the three.
</Callout>

### Reattach on restart

`loadPersisted()` rebuilds the fleet from `state.json`. It re-creates each agent with
`autoRoute: false` (don't re-route a known agent) and `bypassCap: true` (a restored
agent is already accounted for, so it never trips the WIP cap):

- a **`flue-service`** member is restored via its own path (no live process to
  reattach — it is re-onboarded);
- a **`workflow`** run resumes from its persisted `workflowState` checkpoint: if the inner
  omp thread is still alive it reattaches in place, otherwise it is respawned cold and the
  graph continues from the **two-phase checkpoint** (one written at each node's entry, one at
  its exit) — completed stages are **not** re-run and their commits are not duplicated; only a
  node interrupted mid-execution re-runs, bounded by a resume poison cap of 3 cold re-entries
  before escalating to a human. A run with no checkpoint cannot resume; the checkpoint is what
  makes restart safe (OMPSQ-165);
- an **`omp-operator`** is re-created, and because the agent's `agent-host` survived
  the restart, `RpcAgent.start()` simply reconnects to the live socket and replays the
  ring buffer — the agent kept running with full context the whole time.

## The wire protocol

### `SquadEvent` — manager → surface

Everything the manager publishes is one of these. `SquadServer.broadcast` JSON-encodes
each to every WebSocket client (stamping `version` onto `roster`); the TUI consumes the
same objects in-process.

```ts title="src/types.ts"
export type SquadEvent =
  | { type: "roster"; agents: AgentDTO[]; version: string }
  | { type: "agent"; agent: AgentDTO }
  | { type: "removed"; id: string }
  | { type: "transcript"; id: string; entry: TranscriptEntry }
  | { type: "log"; level: "info" | "warn" | "error"; text: string }
  | { type: "commands"; id: string; commands: CommandInfo[] }
  | { type: "features-changed" };
```

A new WebSocket gets a full `roster` snapshot on `open` (plus any per-agent
`commands`). `transcript` replay is unicast to the socket that asked (`subscribe`);
`agent` is the per-agent delta broadcast after every state change.

### `ClientCommand` — surface → manager

Every mutation a surface can request, routed through the single `applyCommand(cmd,
actor)` entry point (`create` and `commission` are handled at the server before
reaching it, because they return a value):

```ts title="src/types.ts"
export type ClientCommand =
  | { type: "prompt"; id: string; message: string }
  | { type: "answer"; id: string; requestId: string; value: string }
  | { type: "interrupt"; id: string }
  | { type: "kill"; id: string }
  | { type: "restart"; id: string }
  | { type: "remove"; id: string; deleteWorktree?: boolean }
  | { type: "create"; options: CreateAgentOptions }
  | { type: "snapshot" }              // request a full roster + recent transcript replay
  | { type: "subscribe"; id: string } // transcript replay of one agent
  | { type: "commission"; spec: CommissionSpec };
```

<Callout type="info">
  `applyCommand` takes an `actor` (default the local operator). It is the seam Phase-2
  federation rides: a remote peer's `{ command, actor }` flows through the *same*
  method, after authorization, so there is no separate remote-command path.
</Callout>
