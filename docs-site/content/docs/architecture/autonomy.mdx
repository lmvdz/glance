## Orchestrator — the self-healing loop

`Orchestrator` (`src/orchestrator.ts`) is a periodic control loop. It is strictly
opt-in: `start()` arms no timer and `tick()` returns immediately unless
`OMP_SQUAD_AUTODRIVE` is set, so the daemon leaks no timer and the loop is fully inert
when off. When armed, it runs every 30s (default `intervalMs`), driving the fleet
through **injected dependencies** (`spawn` / `verify` / `land` / `route`), so the
policy is testable headless with no live daemon.

Each tick applies one policy in order:

1. **Auto-land** — for every idle agent with unlanded work (`featureId`), run the
   acceptance gate; a green gate lands the branches (and the land path closes the
   tracking Plane issue). A land blocked by divergence/conflict is left for the
   conflict path and retried next tick.
2. **Self-heal** — a red gate routes through `routeFailure`: `retry` re-verifies next
   tick under a higher attempt count, `hold` parks it, `escalate` summons a human.
   Attempts are tracked per agent.
3. **Catastrophe** — the *only* place a human is summoned. A clear `CATASTROPHE:` log
   fires on repair-budget exhaustion or an `isCatastrophic` tripwire (infra failure,
   safety violation, regression oscillation), and the loop stops acting on that agent
   until restart. Never a silent drop.
4. **Admission drain** — parked spawn requests are admitted while the WIP cap has
   headroom, re-counting live agents each pass so freshly spawned ones tighten the
   ceiling.

```ts title="src/orchestrator.ts — opt-in arming"
start(intervalMs = 30_000): void {
  if (this.timer || !autodrive()) return;   // autodrive() === !!process.env.OMP_SQUAD_AUTODRIVE
  this.timer = setInterval(() => void this.tick(), intervalMs);
  this.timer.unref?.();
}
```

The orchestrator has two additional guards that prevent runaway or overlapping ticks:

- **Re-entrancy guard** — a `ticking` flag is set at the top of `tick()` and cleared in a
  `finally` block. If the previous tick is still in-flight when the next interval fires, the
  new tick returns immediately rather than overlapping.
- **Bounded ledger purge** — at the end of each tick, the attempt/state ledger is pruned to
  drop entries for branches that no longer appear in the roster, keeping memory bounded.

## Scheduler — WIP cap and FIFO park

`Scheduler` (`src/scheduler.ts`) owns the global concurrency ceiling. The cap counts
**live** agents — everything not `stopped` or `error` (`liveAgents`) — against
`OMP_SQUAD_MAX_WIP` (default 6), read per call so an ops/test change takes effect
without a restart.

When a spawn would exceed the cap, behavior depends on one flag:

- **default** — the historical hard-cap error: the spawn is rejected.
- **`OMP_SQUAD_QUEUE_ON_FULL`** — the spawn is **parked** in a FIFO queue and `create()`
  returns a synthetic DTO with `queued: true` (the signal to the caller) instead of
  erroring. That DTO is never added to the roster; the orchestrator's admission-drain
  step dequeues and spawns the real agent once a slot frees.

```ts title="src/scheduler.ts"
cap(): number { return Number(process.env.OMP_SQUAD_MAX_WIP) || 6; }
canAdmit(liveCount: number): boolean { return liveCount < this.cap(); }
```

## Supervision — answering prompts without a human

There are **two distinct supervisors** with opposite defaults. Knowing which is which
matters.

### Standing supervisor loop — default on

`startSupervisor` (`src/supervisor.ts`) is a standing loop the daemon starts on boot.
It is gated by `OMP_SQUAD_AUTO_SUPERVISE` and runs **unless** you set that to `0` or
pass `--no-supervise`:

```ts title="src/index.ts"
const supervise = process.env.OMP_SQUAD_AUTO_SUPERVISE !== "0" && flags["no-supervise"] !== true;
const stopSupervisor = supervise ? startSupervisor({ port, model: process.env.OMP_SQUAD_SUPERVISE_MODEL || undefined }) : undefined;
```

It opens the daemon WebSocket and, for every agent reporting status `input`, consults a
**one-shot model** for a decision: `decide()` spawns `omp -p --mode json --smol`
(or `--model OMP_SQUAD_SUPERVISE_MODEL`) with the `SUPERVISOR_SYSTEM` prompt, then
submits the answer as an `answer` `ClientCommand`. The system prompt biases toward
**approve / advance** but denies anything destructive or blast-radius-escaping. It is
robust by construction: a non-zero exit, timeout, empty output, or parse failure all
degrade to `chooseFallback(req)` (a deterministic approve/continue).

### In-manager auto-supervise — default off

`maybeAutoSupervise` (`src/squad-manager.ts`) is a separate, **off-by-default**,
deterministic layer gated by `OMP_SQUAD_AUTOSUPERVISE` (note: no underscore). It runs
inside the manager when a request is first registered and answers only what
`chooseFallback` can decide with no model. It is bounded three ways:

- a **risk filter** — `isRiskyRequest` leaves anything matching a destructive pattern
  (`force-push`, `delete`, `deploy`, `prod`, `mainnet`, `secret`, `push…main`, …) for
  a human, and **every host-tool call** is always left for a human (it runs real side
  effects);
- a **per-agent budget** — `OMP_SQUAD_AUTOSUPERVISE_BUDGET` (default 5); past it, that
  agent's requests fall back to the human queue;
- an **audit trail** — every auto-answer (and every skip, with a reason) is logged, and
  the answer is stamped with an `auto-supervise` actor so the transcript shows it
  wasn't a human.

<Callout type="info">
  Both supervisors auto-approve safely *only* because squad agents live in isolated git
  worktrees reviewed before merge. If non-worktree (in-place / production) agents were
  ever supervised, the policy would need tightening — see the `ponytail:` ceiling note
  on `chooseFallback`.
</Callout>

## Resolver — bounded escalation

`routeFailure` (`src/resolver.ts`) is the pure, deterministic decision the orchestrator
reads when a run fails. It maps a failure kind to `retry` / `hold` / `escalate`:

- **`red`** (a gate failed) — retry while attempts are under the repair budget
  (`OMP_SQUAD_REPAIR_BUDGET`, default 3), then escalate;
- **`conflict`** (a land conflicted) — retry exactly once (the automated
  conflict-resolver's single shot), then escalate.

```ts title="src/resolver.ts"
export function routeFailure(kind: FailureKind, ctx?: FailureContext): FailureRoute {
  const attempts = ctx?.attempts ?? 0;
  if (kind === "conflict") return attempts < 1 ? "retry" : "escalate";
  const budget = Number(process.env.OMP_SQUAD_REPAIR_BUDGET) || 3;
  return attempts < budget ? "retry" : "escalate";
}
```

It is only the decision — the orchestrator drives the actual re-attempt, and an
`escalate` becomes the human-summoning `CATASTROPHE:` log. The budget is re-read per
call, so ops can retune it live.

## Scout — per-hour LLM-call budget

The Scout loop extracts opportunities from agent transcripts by calling the LLM on each
new segment. On a large fleet this can be expensive; `OMP_SQUAD_SCOUT_MAX_CALLS_PER_HOUR`
(default 30) caps how many LLM calls the Scout loop may make in any trailing 60-minute window.

When the budget is exhausted for the hour:
- Scout **skips the LLM extraction** for that agent/segment.
- A `warn`-level automation event is emitted: `"scout LLM budget reached (N/h) — skipping extraction for <agent>"`.
- The segment is **not discarded** — Scout will re-harvest it once the trailing-hour window slides and budget is available again.

Set `OMP_SQUAD_SCOUT_MAX_CALLS_PER_HOUR=0` (or any non-positive value) for unlimited calls.

## Background loop observability

All four background loops — **scout**, **observer**, **opportunity**, and **dispatch** — now
surface failures rather than swallowing them silently:

- **Failures are logged** at the `error` or `warn` level and emitted as automation events
  (visible in the **Automation** panel and via `omp-squad automation`).
- **Transient external calls** (Plane API polls) use a **single-retry** pattern: a transient
  failure retries once immediately; if the retry also fails, the error is surfaced as a warning
  rather than silently swallowed.

### Inspecting loop activity

```sh title="CLI"
omp-squad automation            # recent events, all loops
omp-squad automation --loop scout --window 2h
```

```sh title="API"
GET /api/automation
GET /api/automation?loop=scout&windowMs=7200000&meaningful=1
```

The response includes:
- `events` — the raw event feed (timestamped, per-loop, with level and detail).
- `rollup` — per-loop aggregates: `success`, `error`, `skip`, `llmCalls`, `durationMs`.

The **Automation** panel in the web dashboard visualizes this feed and refreshes on demand.
