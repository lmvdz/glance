A workflow turns an agent's *process* into a reviewable artifact. Instead of handing one
free-text task to a model and trusting whatever it decides, you author the process as a
directed graph â€” plan, human-approval gate, implement, verify, bounded fix-up loop â€” in the
Graphviz/DOT subset that [fabro](https://github.com/fabro-sh/fabro) uses. The graph is a
diffable, version-controlled file (`.fabro`). You review the *plan* before the diff and
intervene only at the gates.

## A workflow is just another agent

omp-squad's architecture is "anything implementing `AgentDriver` joins the roster, status
derivation, transcript, TUI, web, federation, and Plane spawn" â€” the same contract that
`RpcAgent` and `FlueServiceDriver` satisfy. `WorkflowDriver` (`src/workflow-driver.ts`) is
that contract again, so `kind: "workflow"` lands a graph-driven, gated, multi-model-ready run
in exactly the same surfaces as an omp operator. See [the driver seam](/docs/architecture/agent-driver).

| `AgentDriver` member | Workflow meaning |
|---|---|
| `start()` | Parse the graph (`workflowPath`) or take a pre-built one (`workflow`); build the engine + executor. |
| `prompt(goal)` | The **first** prompt starts the run with that goal; later prompts steer the live inner agent. |
| `abort()` | Stop the engine at the next node boundary and abort the inner turn. |
| `getState()` | Synthetic â€” the executor's stage rollup becomes `todoPhases` (the dashboard's done/total). |
| `respondUi()` | Answer a human gate, or pass through the inner agent's own approval request. |
| `detach()` | Leave the inner host running across a daemon restart. |
| `event` frames | One `agent_start â€¦ agent_end` around the whole run; stage transitions + command output between. |

The manager footprint is tiny: a `kind`, one `makeDriver` branch, one poll-filter widen
(`src/squad-manager.ts`) â€” identical to how `flue-service` dropped in.

## Pure engine, injected execution

The engine is **pure orchestration** and execution is an injected seam. This is the decision
that lets a fan-out spawn *real roster agents* instead of hiding branches inside one row.

- **`WorkflowEngine`** (`engine.ts`) knows nothing about omp, processes, or the fleet. It
  walks the graph: routing, edge conditions, visit caps, goal-gate failure handling,
  human-gate pausing, and stage emission. It is unit-tested against a fake executor (no tokens).
- **`NodeExecutor`** (the seam in `types.ts`) decides what "run an agent node", "run a
  command", or "raise a human gate" concretely *means*. The engine drives it.
- **`SingleAgentExecutor`** (`executor.ts`) is the Phase-A executor. It binds **every agent
  node to one persistent omp thread**, so a whole `plan â†’ implement â†’ verify` run is *one
  steerable roster entry*. Command nodes shell out in the worktree; human gates raise a
  callback; agent/prompt nodes share that single thread (the goal is primed once, command
  output is fed forward).

Because the engine is pure, a later executor can fan agent nodes out to real fleet agents
without touching the driver or the engine â€” which is exactly what [fan-out](/docs/workflows/fan-out) does.

## Stages drive the dashboard todo rollup

Each node execution is a **stage**. The engine emits a `StageEvent` at every node's start and
end; the executor's `onStage` keeps a `rollup` array (`{ label, status }`, `in_progress` â†’
`completed`). `WorkflowDriver.getState()` maps that rollup into `todoPhases`, so the run shows
a live done/total exactly like an omp operator's todo list â€” no special workflow UI. The
inner agent's per-turn `agent_start`/`agent_end` frames are swallowed so the run reads as one
continuous working session.

<Callout type="info">
Workflow runs persist a resumable checkpoint at every node boundary (`WorkflowRunState` =
engine checkpoint + rollup). After a daemon restart a run resumes from that checkpoint: if its
inner thread is still alive it reattaches in place, otherwise it is respawned and the graph
continues from the last completed node â€” completed stages are not re-run. Only a run with **no**
checkpoint is unrecoverable; the manager logs a warning and does not restart it from scratch.
</Callout>
