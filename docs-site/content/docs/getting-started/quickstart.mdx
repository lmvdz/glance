This walks the full loop: start the daemon, spawn an agent, watch it, steer it,
and tear it down. It assumes `glance` is on your `PATH` (otherwise read every
command as `bun src/index.ts <cmd>` — see [Installation](/docs/getting-started/installation)).

<Steps>
<Step>
### Start the daemon

`glance up` starts one long-running process that opens the TUI and serves the
web dashboard (default port `7878`).

```bash
glance up
```

On a server (or any time you only want the web UI), run it headless:

```bash
glance up --no-tui
```

<Callout type="info">
The daemon generates a bearer access token on first run and prints it on boot
with one-tap sign-in links. Every API and WebSocket request carries it — see
[Remote access](/docs/guides/remote-access).
</Callout>
</Step>

<Step>
### Spawn an agent

From another shell, `add` cuts a fresh worktree of a repo and launches an agent in
it. `--task` is the instruction sent once the agent is ready.

```bash
glance add ~/code/myproject --name auth-refactor \
  --task "Refactor the auth module to use the new session API."
```

It prints the spawned agent's name, status, generated `id`, and worktree path. The
worktree lands on branch `squad/<name>` by default. For every `add` flag
(`--model`, `--approval`, `--thinking`, `--workflow`, `--verify`, `--sandbox`, …)
see [Spawning agents](/docs/guides/spawning).
</Step>

<Step>
### See the roster

```bash
glance list
```

One row per agent: status, name, branch, a `⛔N` badge when it's blocked on input,
and its current activity. (`glance ls` is an alias.)
</Step>

<Step>
### Send a follow-up

`prompt` steers a running agent — either a new instruction or an answer to a
pending request. Use the `id` printed by `add` (or shown in `list`).

```bash
glance prompt auth-refactor-<id> "Also update the tests."
```

(`glance say` is an alias.)
</Step>

<Step>
### Open the dashboard

```bash
glance open
```

This prints the dashboard URL (default `http://127.0.0.1:7878`). The **+ Add
agent** button and per-agent composer / answer controls do everything the CLI
does — see the [web dashboard guide](/docs/guides/web-dashboard).
</Step>

<Step>
### Remove an agent

```bash
glance rm auth-refactor-<id> --delete-worktree
```

Without `--delete-worktree` the agent is removed from the roster but its worktree
is left on disk. (`glance remove` is an alias.)
</Step>
</Steps>

## End-to-end smoke test

This drives a real model from spawn to artifact — it confirms the whole pipeline
works, not just that the suite passes. It spawns a `yolo`-approval agent with a
one-shot task and checks the file it was told to write.

```bash
glance up --no-tui &
glance add /path/to/git/repo --name demo --approval yolo \
  --task 'Create a file proof.txt containing OK via a shell command, then stop.'
glance list           # watch demo go working → idle
cat ~/.glance/worktrees/<repo>-squad-demo/proof.txt   # → OK  (legacy installs: ~/.omp/squad/worktrees/…)
```

When `list` shows `demo` settle from `working` to `idle` and the `cat` prints
`OK`, the full loop — daemon, worktree, RPC child, task dispatch, status
derivation — is healthy.

<Callout type="warn">
`--approval yolo` lets the agent run shell commands without asking. It's safe here
because the agent works in an isolated, reviewed-before-merge worktree — but only
use it deliberately.
</Callout>
