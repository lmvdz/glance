glance coordinates agents on **one** machine through the shared registries in the
state dir (default `~/.glance`, legacy `~/.omp/squad`; presence + leases, one file per
claim, heartbeat-TTL). Federation promotes
that coordination **across machines**, so two operators working the same repo from different
hosts see each other's agents and file leases.

```text
 host A                         coordinator                    host B
 ┌───────────────┐    ws        ┌───────────┐      ws       ┌───────────────┐
 │ glance up  │─presence────▶│  relay    │◀────presence──│ glance up  │
 │ federation-   │─leases──────▶│ (fan-out) │─────leases───▶│ federation-   │
 │   sync        │◀─leases──────│           │◀────leases────│   sync        │
 └───────────────┘              └───────────┘               └───────────────┘
```

<Callout type="info">
**Federation is on by default.** The daemon starts with a `LocalFederationBus` — a real bus that
works on a single host with zero config (local pub/sub, own roster, collision detection). It
gossips to peers only once `GLANCE_COORDINATOR` is set. `GLANCE_FEDERATION=0` opts all the
way out, reverting to the inert `NullFederationBus`.
</Callout>

## Single-host operation (zero config)

`LocalFederationBus` is the default. With no coordinator URL it runs entirely locally:
- Presence is published to a local loopback (the roster updates immediately).
- File leases are tracked in the local registry.
- Collision detection runs over the local roster.
- `/api/federation`, `/api/leases`, and `/api/fabric` return real local data (operator roster,
  live leases, infrastructure summary).

No coordinator means no peers, no cross-host gossip — but everything that runs on one machine
works fully, including collision detection between agents on that machine.

```sh title="disable federation entirely"
GLANCE_FEDERATION=0 glance up
# → federation: disabled (GLANCE_FEDERATION=0)
```

## Joining a team room (cross-host)

Set `GLANCE_COORDINATOR` to a coordinator WebSocket URL. The daemon's `LocalFederationBus`
creates an inner `TailnetFederationBus` that connects in the background:

  type={{

    },

```sh title="join a team room"
GLANCE_COORDINATOR=ws://coordinator-host:7900 \
GLANCE_OPERATOR=alice \
glance up
# → federation: joined ws://coordinator-host:7900 as alice
```

The operator identity is `GLANCE_OPERATOR || os.userInfo().username || "local"`, with
`origin: "local"`. The same resolution is mirrored in the server (for labelling the local
roster) and in `federation-sync`, so every surface agrees on who "self" is.

<Callout type="warn">
True cross-host federation requires three things not available on a single workstation: (1) a
**deployed coordinator** reachable by all hosts, (2) a **second physical host** running
`glance up` pointed at the same coordinator, and (3) a **Tailscale tailnet** for remote-operator
identity — `tailscale whois` only resolves on a real tailnet; on localhost it falls back to `"local"`.
The rest of this page explains what works without them.
</Callout>

## The coordinator

`src/coordinator.ts` is a **dumb, protocol-agnostic WebSocket relay**: every frame received
from one client is rebroadcast verbatim to every *other* client. It never parses frames, so
the wire protocol stays owned entirely by the buses (`TailnetFederationBus`). A plain `GET`
gets a tiny `glance coordinator` health body; the upgrade is accepted on any path.

The relay **binds `127.0.0.1` by default** and carries a **pre-shared-token auth gate** —
without it any peer that can reach the port snoops every presence/lease frame and can
spoof/impersonate. Exposing it (`GLANCE_COORDINATOR_HOST=0.0.0.0`) therefore requires
`GLANCE_COORDINATOR_TOKEN` on the coordinator and on every client; a non-loopback bind
with no token is refused unless `GLANCE_INSECURE=1`. The token rides the WS handshake as
the `ompsq-token` subprotocol, exactly like the daemon's bearer token.

Run one anywhere on the tailnet via `src/coordinator-main.ts`:

```sh title="run a coordinator"
GLANCE_COORDINATOR_HOST=0.0.0.0 \
GLANCE_COORDINATOR_TOKEN=$(openssl rand -hex 32) \
GLANCE_COORDINATOR_PORT=7900 bun src/coordinator-main.ts
# → glance coordinator listening on ws://127.0.0.1:7900 (token-gated)
```

Put it on a Tailscale node and point every operator at its tailnet address. WireGuard
encrypts the wire; Tailscale ACLs gate who can reach it; identity is the tailnet SSO login,
resolved via `tailscale whois --json <ip>` (`tailscaleWhois` in `federation.ts`). The bus
auto-reconnects with capped backoff (500 ms → 30 s) and **never throws a transport error** to
the caller — an unreachable coordinator simply means an empty, silent room.

## Transport rationale: why a relay, not a P2P stack like iroh

The transport is deliberately minimal: **Tailscale provides the secure network, identity, and
NAT traversal; the coordinator provides fan-out.** Everything sensitive — encryption, who-is-who,
reaching a host behind NAT — is delegated to the tailnet, leaving glance with the ~85-line
opaque relay above plus a gossip bus.

A natural question is whether to replace this with a peer-to-peer networking stack such as
[iroh](https://github.com/n0-computer/iroh) — "dial by public key", QUIC with hole-punching and
public-relay fallback, plus `iroh-gossip` (pub/sub) and `iroh-docs` (an eventually-consistent
key-value store). The conceptual fit is real: `iroh-gossip` maps onto the presence/lease gossip,
and `iroh-docs` onto the lease registry.

<Callout type="info">
**Decision (2026-06): stay on Tailscale + the WS relay; do not adopt iroh.** iroh would *replace*
the things the tailnet already provides, not add a missing capability — and it is a Rust library.
Pulling it into this TS/Bun, zero-native-dependency codebase (see the [Contributing](/docs/contributing)
ponytail mandate) means an FFI addon (`iroh-ffi`) or a Rust sidecar: a large footprint for a
transport that already works and is opt-in.
</Callout>

What each layer would map to, if the substitution were ever made:

| glance today | iroh equivalent |
|---|---|
| Tailscale tailnet (encryption, NAT traversal, identity) | iroh endpoint — public-key identity, hole-punch + relay fallback, QUIC encryption |
| `coordinator.ts` WS relay (central fan-out) | `iroh-gossip` (serverless pub/sub overlay) |
| `presence` / `leases` gossip frames | `iroh-gossip` topics, or `iroh-docs` for lease/presence state |
| `tailscale whois` → `Actor` | iroh `EndpointId` (public key) — no tailnet SSO |

**When to revisit:** only if the tailnet requirement itself becomes the limitation — i.e. you need
to federate operators who are *not* on a shared tailnet, with zero infrastructure (no coordinator to
host) and zero-config NAT traversal. Even then the clean integration is a **sidecar** (an `iroh`
process the daemon speaks to) behind the existing `FederationBus` seam in `src/federation.ts` — not
an in-process native dependency — so `NullFederationBus` / `TailnetFederationBus` would simply gain
an `IrohFederationBus` sibling.

## Presence

The squad daemon publishes its `OperatorPresence` to the coordinator through its bus, on every
roster change. A presence frame is the operator's `Actor`, an `Availability`, the host, the
live `AgentDTO[]`, and a timestamp.

```ts title="src/types.ts"
export type Availability = "active" | "away" | "offline";

export interface Actor {
  id: string;            // tailnet login "bob@company.com" or "local"
  displayName?: string;
  origin: "local" | "remote";
}

export interface OperatorPresence {
  operator: Actor;
  availability: Availability;
  host?: string;
  agents: AgentDTO[];
  updatedAt: number;
}
```

Two pure primitives fold peer rosters together:

- `mergeRosters(self, peers)` — dedupes by `operator.id`, newest `updatedAt` wins, and pins
  `self` to the head of the list.
- `detectCollisions(presences)` — flags repos where agents owned by **different** operators
  share the same branch, keyed on the **normalized `repoId`** (see [Cross-host repo identity](#cross-host-repo-identity) below) — not the host-local path.

This means:
- Two hosts that cloned the **same origin at different local paths** still produce a collision — the `repoId` matches.
- Two agents working on repos that **happen to share a directory name** but have different origins do NOT produce a false collision.
- Each `AgentDTO` carries a `repoId` field stamped by `LocalFederationBus` before the presence frame leaves the host.

`federationView` composes them; `PeerRoster` collects the newest presence frame per remote
operator and prunes any peer that stops gossiping past the 90-second TTL
(`PEER_PRESENCE_TTL_MS`).

### Surfacing remote presence

The server exposes `GET /api/federation` (bearer-gated like the rest), returning a
`FederationSnapshot`:

```ts
{ coordinator: string | null, operators: OperatorPresence[], collisions: Collision[] }
```

`self` (this host's live roster) is always included. With no coordinator the `coordinator`
field is `null`, the `operators` array contains only the local operator, and the panel in
the web dashboard stays hidden. The endpoint never errors in either mode — it always returns
real local data. In DB mode the roster is per-org scoped so one org never sees another's agents.

The **web dashboard** renders this as the **Federation** panel in the project view: peer
operators, their agents, and any shared-branch collisions. With a coordinator configured and
a second host connected, cross-host collisions surface here.

## Cross-host file leasing

Leasing already prevents two agents on **one** host from clobbering the same file (advisory
soft-block-with-override; `LEASE_TTL_MS` = 120 s). The `federation-sync` companion
(`src/federation-sync.ts`) extends that across hosts — **decoupled from the daemon**, working
purely off the on-disk registries in the state dir:

<Steps>
<Step>
**Publish** — every tick (`PUBLISH_INTERVAL_MS` = 15 s) it reads *this operator's own* live
leases per repo and gossips them, keyed by the repo's **cross-host identity**. It never
re-broadcasts leases it mirrored from a peer.
</Step>
<Step>
**Mirror** — a peer's leases for a repo you *also* have locally are written into your local
lease registry (`mirrorLease`, preserving the remote operator/host). From there everything
just works: the lease-hook's `holdersOf` warns when you edit a file a remote teammate holds,
and the command center's **Files in flight** panel shows it. The TTL prunes a mirrored lease
once the peer stops gossiping.
</Step>
</Steps>

Run it as its own process:

```sh title="run federation-sync"
GLANCE_COORDINATOR=ws://coordinator-host:7900 \
GLANCE_OPERATOR=alice \
GLANCE_FED_REPOS=/home/alice/app,/home/alice/infra \
bun src/federation-sync-main.ts
```

<TypeTable
  type={{
    GLANCE_FED_REPOS: {
      type: 'comma-separated paths',
      description: 'Extra repo paths to always gossip leases for, beyond the cwd repo and any repo discovered in the local presence registry.',
    },
  }}
/>

It watches the current working directory plus `GLANCE_FED_REPOS`, plus every repo
discovered in the local presence registry. `GLANCE_COORDINATOR` is **required** — without
it the process prints a hint and exits.

### Cross-host repo identity

`repoIdentity()` keys federation on the **normalized git origin URL**, not a host-local path,
so two hosts that cloned the same origin agree:

```text
git@github.com:acme/app.git
https://github.com/acme/app          ─▶  github.com/acme/app
ssh://git@github.com/acme/app.git
```

It reads `git config --get remote.origin.url` (the raw configured URL, **not**
`git remote get-url`, which expands per-machine `insteadOf` rewrites). A repo with no origin
falls back to `name:<basename>` — best-effort, advisory only.

## API surface

The federation endpoints are real and available in both single-host and cross-host modes:

| Route | Returns |
|---|---|
| `GET /api/federation` | `{ coordinator, operators, collisions }` — merged local+peer roster, cross-host collisions keyed on `repoId`. |
| `GET /api/leases?repo=<path>` | Live file leases for the requested repo (or cwd). Per-org scoped in DB mode. |
| `GET /api/fabric?repo=<path>` | Infra summary: roster, leases, and health for the requested repo. Always includes leases. |
| `POST /api/federation/command` | Send an addressed remote command to a peer (see below). Operator-tier; audited as `federation.command`. |
| `GET /api/federation/capabilities` | Capability-pack **metadata** advertised to peers — names/versions only, until explicit context policy allows more. |

## Remote steering — commands across hosts

Both halves of "drive a teammate's agent from your command center" exist:

- **Send** — `POST /api/federation/command` with `{ to: "<operator id>", cmd: { type, … } }`
  publishes an addressed `RemoteCommand` over the bus. Frames are addressed, so peers drop
  commands not meant for them.
- **Receive** — the peer daemon's `onRemoteCommand` re-authorizes every inbound command with
  its *own* RBAC before applying it via `applyCommand(cmd, actor)`. The coordinator stamps
  the sender's real socket address as `ip` (any client-sent `ip` is overwritten), so the
  receiver's `tailscale whois` verifies the true sender — never a claimed identity. An
  unverified sender resolves to the read-only `viewer` tier: sending grants nothing.

Still missing for a full remote-steering UX: a command-center UI affordance on the Federation
panel (the API is the seam) and a reply/ack channel — effects surface through the peer's
gossiped presence.
