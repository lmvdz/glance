# Design: never-lose-work

_Origin: /research of Apache Burr → plans/research-burr/BRIEF.md (patterns 1, 5). Adversarial design 2026-07-03: sonnet designer → 2× fable red team → fable arbiter._

## Approach
The slice ships two coupled mechanisms. **§1 Fork-from-checkpoint**: an append-only per-run checkpoint log (`src/workflow/checkpoint-log.ts`, receipts-shaped JSONL under `stateDir/workflow-checkpoints/<runId>.jsonl`) written from the existing single `checkpoint` listener in squad-manager (engine node-boundary checkpoints only — branch-done emissions carry a `transient` flag and are excluded), with `seq` = a per-runId serialized append counter initialized from the file's line count on first append after boot (mirrors `writeChain` promise-chaining; read path sorts by seq and skips torn trailing lines). Each entry best-effort captures `headSha` (data only — no rewind behavior this slice). A new engine-level terminal-failure channel replaces the draft's two-site wiring: `opts.escalate(reason, checkpoint)` is invoked at ALL four terminal-failure returns in `engine.ts` (visit-cap-no-overflow at 80, poison cap at 94, failed-no-recovery-route at 118 — which also covers noProgressRoute exhaustion, the most common verify-cascade death — and ran-off-the-end at 128). The driver forwards it as a `workflow_terminal` event; the manager persists `workflowState.terminal = {reason, at, forkPoint:{runId,seq}}`, then escalates through the EXISTING `markCatastrophe` channel (sticky "error" + attention queue + push) rather than a new ad-hoc status write. `terminal` is the load-bearing lifecycle bit: the `resumable` predicate (squad-manager.ts:789), `reconnectLive`'s auto-resume, `makeDriver`'s `resumeState`, and adoption all exclude terminal-marked runs, killing the boot-loop where a poisoned run re-adopts, re-trips the cap, and burns a ceiling slot every restart. `dto.forkAvailable` is derived from the persisted marker (survives restarts). `fork(id, {seq?})` then: resolves base as `git rev-parse HEAD` in the original worktree (never `dto.branch`, which is undefined for in-place/ad-hoc agents; refuse with a clear error if the worktree is gone); creates the ref via `git branch squad/<newId> <sha>` and reuses UNMODIFIED `addWorktree` (existing-branch checkout path — no `fromRef` extension); validates the chosen checkpoint's `currentNode` against the parsed graph; seeds `WorkflowRunState` with `forkedFrom:{runId,seq}`, `resumeAttempts:0`, `rollup:[]`, and — the decision that makes fork actually work for the escalate-exhaustion case — **resets `visits` for every fix-up tier** (each goalGate's `retryTarget` plus its `overflow` closure) while carrying all other visit counts forward; marks the original `terminal.supersededBy = newId` (excluded from adoption and dispatch; fork inherits the issue), clears the fork offer, and spawns via an internal-only create path with `bypassCap:true` (operator-initiated, rare; sidesteps the queue-on-full re-entry problem), `cold:true`, name stabilized by stripping any existing `-fork` suffix before appending one. Fork is offer-only (never auto-fired), refused while the original is `working`, and limited to one live fork per source runId.

**§2 Deterministic fan-out identity**: the branch key is `${nodeId}#${visitIndex}:${branchIndex}` — deliberately runId-FREE inside the state so recorded outcomes survive a fork — while the branch AGENT id is `br-<hash8(runId + ":" + key)>-<slug(nodeId,12)>`: deterministic, collision-free across runs/forks, and short enough that `socketPathFor(id)` stays under the 108-byte sun_path limit (the full key travels as a DTO/persisted field, not in the id). The engine constructs `visitIndex`/`branchIndex` in `runParallel` and threads the key through a fourth `runBranch` arg (engine → SingleAgentExecutor.runBranch → BranchSpec.branchKey → spawnFleetBranch); the runId prefix for the agent id is applied in the driver, which owns runId. `spawnFleetBranch` passes the derived id through a **private internal create path** (`createInternal`, sharing `create()`'s body via an internal-only options type) — `explicitId` NEVER appears on `CreateAgentOptions`, because `server.ts:1358` spreads wire-supplied options straight into `create()` and any type-level field is attacker-reachable; `createInternal` additionally rejects when `this.agents.has(id)`. It also persists the branch task (`opts.task`) so a branch is re-promptable after restart. Per-branch completion: `WorkflowRunState.branchOutcomes?: Record<key, {disposition: "succeeded"|"failed"|"not_attempted"; text?: string; at: number}>` where ceiling/WIP refusals, abort teardowns, and spawn-crashes record `not_attempted` (re-spawnable on resume) and only genuinely-executed terminal results (including the 30-min timeout) record `succeeded|failed`. Emission: `runParallel` snapshots the fork node's ENTRY checkpoint (its `resumeAttempts`, `visits` before branch increments, `currentNode`, `index`) once at fan-out start; each `onBranchDone` emits that snapshot verbatim plus the full accumulated `branchOutcomes` map, flagged `transient`. Because the engine always emits the complete map inside an entry-snapshot clone, the manager's existing overwrite listener needs NO merge logic, the poison counter can never be reset by a mid-fan-out emission, no branch-visit increments leak into persisted state, and the map self-clears the moment the merge node's ordinary exit checkpoint (which carries no `branchOutcomes`) overwrites it. Cold resume of a parallel node: skip branches with `succeeded|failed` dispositions (use recorded outcomes in the join); for `first_success`, short-circuit immediately if any recorded disposition is `succeeded`; re-spawn `not_attempted`/absent keys under the same deterministic ids (worktree reuse via `addWorktree`'s registered-path idempotency); and — the reconciliation protocol, specified rather than hand-waved — enumerate all expected keys, and for any live roster agent whose id matches a key the resume will NOT run (resolved or short-circuited) OR a key it WILL run, `stop()` it first; re-spawned branches are re-prompted with their persisted task in the reused worktree ("stop-and-reprompt" — no await-in-flight protocol; the pre-crash turn's collector and buffer are gone, so its text is unrecoverable by construction and the recorded-disposition map is the only authority). The draft's `deferredResumable` extension is CUT: under this model the checkpoint is the source of truth, worktree reuse comes from the deterministic id, and the extension was proven both insufficient (adopted-parent case still drops children) and unnecessary. Two adjacent guard holes get closed in passing: `restart()` passes `cold:true` for workflow agents (it always rebuilds a fresh inner thread, so today it silently bypasses the poison cap), and `applyCommand` gets a default-case error so an old daemon rejects an unknown `fork` command instead of no-oping (webapp additionally gates the button on `dto.forkAvailable`).

Surfaces and sequencing: `ClientCommand` gains `{type:"fork"; id; seq?}`; a read-only `GET /api/agents/:id/checkpoints` returns `[{seq, at, currentNode, outcome}]` (never `vars`); the webapp TaskDetail adds "Fork from step N" beside Restart — default latest, earlier steps labeled "routing state only — code stays at the branch tip" (Candidate-A semantics; rewind is future work enabled by the captured `headSha`). Checkpoint-log hygiene: `vars.lastOutput`/`lastText` truncated to 4KB per entry; the run's JSONL deleted on `remove(deleteWorktree:true)`. Build order: §2's engine callback plumbing (`onBranchDone`/escalate-with-checkpoint threading through `run()`'s opts) first, then §1's log/fork/UI on top of it. Tests (bun test, node_modules/.bin on PATH): seq monotonicity across a simulated restart; poison cap still trips at 3 under interleaved branch-done emissions; `maxVisits:1` branch survives a resume; first_success resume with a recorded winner spawns nothing and stops reattached losers; fork of an escalate-exhausted run re-enters the fix-up tier with reset visits; terminal-marked run is not re-adopted and not auto-resumed by reconnect; wire-level `create` with an injected id-shaped option is rejected.

## Key Decisions
| Decision | Choice | Alternatives considered | Rationale |
|---|---|---|---|
| How fork treats retry budgets (the visits carry-forward contradiction both red teams ranked critical) | Reset visits ONLY for fix-up tiers (each goalGate's retryTarget + its overflow closure) and resumeAttempts to 0; carry all other visit counts forward | Full carry-forward (draft — makes fork a near-no-op for escalate exhaustion, one pass then dead); full reset (unbounds runaway re-fork loops); UI-defaults-to-earlier-checkpoint (shifts a correctness decision onto the operator) | Both red teams independently proved carry-forward buys exactly one node re-execution for the visit-cap dead end (engine.ts:71 skips the cap once, then codefix/fixup/escalate are all cap-blocked). Resetting exactly the tier chain is deterministic, computable from the parsed graph, refreshes the budget the operator is explicitly asking to retry, and keeps global/structural caps as the anti-runaway bound. This also keeps fork-from-latest genuinely useful, so the checkpoint log is history/lineage, not a correctness crutch. |
| How deterministic ids are injected into create() without opening the wire-level id-collision hole | Private createInternal path with an internal-only options type; explicitId never exists on CreateAgentOptions; createInternal rejects duplicate ids; both spawnFleetBranch and fork() use it | explicitId on CreateAgentOptions guarded by parentId-present (draft — attacker-satisfiable through server.ts:1358's {...cmd.options} spread, and self-contradictory since forks have no parentId); stripping the field in the server handler (fragile, every future wire entry point must remember) | server.ts:1358 spreads client-supplied options straight into create(), so any type-level field is network-reachable; a forged id silently replaces the victim's AgentRecord (agents.set at 2151) and checks out its worktree. Keeping the capability off the public type makes the guard structural. | Terminal-failure lifecycle of the original run (poison cap, cap exhaustion, no-progress, off-the-end) | Persist workflowState.terminal={reason,at,forkPoint,supersededBy?}; exclude terminal from resumable/reconnect-auto-resume/makeDriver resumeState/dispatch; escalate via the existing markCatastrophe channel; derive dto.forkAvailable from the marker | DTO-only forkAvailable + status flip (draft — lost on restart, and the poisoned run boot-loops through adoption burning ceiling slots, or worse warm-reattaches with NO poison cap since the cap only applies when resume.cold); leaving failed runs idle for the orchestrator (today's behavior — silent, and re-verifies an exhausted cascade toward the same failure) | Verified: resumable is just kind===workflow && workflowState!==undefined (789), reconnectLive auto-resumes any live-thread workflow, and markCatastrophe (1730) is the established sticky-error + attention-queue + push precedent. The entry-snapshot-clone + full-map emission resolves three findings at once with zero manager-listener changes: the poison counter can't be reset mid-fan-out (same resumeAttempts as entry), branch-node visit increments never persist (RT2's maxVisits:1 skip-as-failed bug), and the map self-clears when the merge node's ordinary checkpoint overwrites it. Timeout records as failed, not not_attempted: a branch that consumed its 30-minute budget will likely do so again; auto-respawning it on every resume is a loop, and the operator can fork if it deserves a retry. |
| Branch key scheme vs agent id | State map keyed runId-free (`${nodeId}#${visitIndex}:${branchIndex}`); agent id = `br-<hash8(runId+key)>-<slug(nodeId,12)>`; full key carried as a DTO/persisted field | runId-embedded key used verbatim as agent id (draft — severs outcome lineage across a fork since forks mint new runIds, and composes an unbounded string that flows into socketPathFor and can exceed the ~108-byte sun_path limit, presenting as host-never-alive) | Determinism requires stability, not readability. The runId-free map key keeps a branch's identity stable across a resume of the SAME run (the fork node's visit count doesn't change mid-fan-out); the runId-salted agent id keeps worktrees/sockets collision-free across runs and forks. Short hashed ids stay socket-safe with any nodeId length and avoid `:`/`#` traveling into socket filenames, receipts paths, and branch names. |

## Risks
- Stop-and-reprompt reconciliation discards a live branch's in-flight turn: any un-committed reasoning in that turn is lost and the task re-executes in the same worktree. Bounded by worktree persistence (prior file state survives) and by the fact that the lost text was unrecoverable under any design; watch for prompts that are not re-entrant (mitigate in the re-prompt template: 'continue; prior partial work may exist in this worktree').
- Terminal-marking removes exhausted workflow runs from the orchestrator's idle-pickup repair loop. For genuinely exhausted cascades this is correct, but if a class of failures emerges where the orchestrator's independent repair would have succeeded, outcomes regress to needs-human. Mitigation: terminal is set only on the four engine terminal-failure returns, not on every failed workflow_done; monitor via the catastrophe audit trail.
- Fork uses bypassCap:true, so an operator can fork past the WIP cap; the hard agent ceiling does not gate this path. Accepted for a rare, deliberate, offer-gated manual action with a one-live-fork-per-run limit; revisit if auto-fork is ever proposed (it is explicitly out of scope).
- The runId-free branchOutcomes key assumes a parallel node's edge order is stable across .fabro re-parses; an edited workflow file that reorders branch edges between crash and resume mis-attributes recorded outcomes to the wrong branch. Same drift class as fork's currentNode validation — cheap hardening later is recording the target nodeId alongside the disposition and skipping mismatches.
- Checkpoint JSONL grows per run despite 4KB truncation (two entries per node boundary, 50-visit default caps ⇒ low-MB worst case per run) and files for never-removed runs accrue; GC is tied to remove(deleteWorktree:true) only. Acceptable now; a stateDir sweep for runIds absent from the roster is a follow-on.
- Deterministic branch ids make worktree reuse automatic, but a re-spawned branch inherits whatever half-done state the crashed attempt left (possibly broken tree, un-staged conflicts). The verify gate downstream bounds the damage, but a poisoned branch worktree could fail repeatedly until the parent's own caps trip.
- Daemon/webapp version skew during rollout: an old webapp never shows fork (no forkAvailable), a new webapp against an old daemon hides the button; but a new daemon adopting OLD persisted records mid-fan-out degrades to full re-fan-out with fresh ids (documented, silent). One legacy generation of orphaned branch worktrees may need manual cleanup.
