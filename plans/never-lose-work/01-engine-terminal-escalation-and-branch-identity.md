# Engine terminal-escalation channel + deterministic branch keys + entry-snapshot branchOutcomes
STATUS: closed
PRIORITY: p0
REPOS: omp-squad
COMPLEXITY: architectural
TOUCHES: src/workflow/engine.ts, src/workflow/types.ts, src/workflow/executor.ts, src/workflow-driver.ts

## Goal
Replace the engine's single-site escalate(reason) with escalate(reason, checkpoint) fired at all four terminal-failure returns (visit-cap-no-overflow, poison-cap, failed-no-recovery-route, ran-off-the-end); give runParallel deterministic per-branch keys and a three-state disposition model, emitting branchOutcomes as a verbatim clone of the fan-out node's ENTRY checkpoint plus the accumulated map, flagged transient; make runParallel resume-aware so cold resume of a parallel node skips branches with recorded succeeded/failed dispositions.

## Approach
**engine.ts**
1. Change `escalate?: (reason: string) => void | Promise<void>` to `escalate?: (reason: string, checkpoint: EngineCheckpoint) => void | Promise<void>` in the `run()` opts type (types.ts NodeExecutor stays untouched — this is a `run()`-local opts field, not on NodeExecutor).
2. Add a private `terminalFail(shared, checkpoint, outcome, reason)` helper that returns the `RunResult` and, if `outcome === "failed"`, awaits `opts?.escalate?.(reason, checkpoint)` first. Build `checkpoint` from the current `ctx`/`shared` state at each site (same shape as the entry checkpoint already built at line ~101: `{goal, currentNode: current, visits: {...shared.visits}, vars: {...ctx.vars}, outcome: ctx.outcome, preferredLabel: ctx.preferredLabel, index: shared.index, resumeAttempts: entryAttempts}`).
3. Wire the helper at all four sites: the visit-cap-no-overflow return (current line 80), the poison-cap return (line 94 — this one already computes `reason` and calls `escalate`, just add the checkpoint arg), the failed-no-recovery-route return (line 118 — covers noProgressRoute exhaustion, the most common dead end), and the ran-off-the-end return (line 128).
4. `runParallel(fork, ctx, shared)` gains a `resume?: {branchOutcomes?: Record<string, {disposition: string}>}` and needs the run's `visits[fork.id]` (already tracked) as `visitIndex`. Compute `branchKey = \`${fork.id}#${visits[fork.id] ?? 0}:${branchIndex}\`` for each branch (branchIndex = its position in `branchIds`).
5. Capture `const entrySnapshot: EngineCheckpoint = {...}` ONCE at the top of `runParallel`, before any branch runs (same shape as the node's entry checkpoint already emitted by `run()` just before calling `runParallel` — reuse `shared`/`ctx` at that instant).
6. Build `const branchOutcomes: Record<string, {disposition: "succeeded"|"failed"|"not_attempted"; text?: string; at: number}> = {}` seeded with `not_attempted` for every branch key.
7. In `runOne`, classify: ceiling/WIP refusal (the `NodeResult` a branch executor returns before ever really running — detect via a convention, e.g. the branch spawner is expected to reject/return a sentinel `{outcome:"failed", notAttempted:true}` — see concern 02 for spawnFleetBranch's contract; add a `notAttempted?: boolean` field to `NodeResult` in types.ts) and the `catch` block (spawn crash / abort teardown) → `disposition: "not_attempted"`. A genuinely-executed result (including the 30-min turn timeout, which surfaces as a normal `{outcome:"failed"}` from the executor) → `disposition: r.outcome` (succeeded/failed).
8. On resume (cold resume of a parallel node — the currentNode IS the fork node and `resume.branchOutcomes` is present), skip branches whose recorded disposition is `succeeded`/`failed`: for `wait_all`, treat them as already-resolved (fold their recorded outcome straight into `results[idx]`, don't call `runOne`); for `first_success`, short-circuit immediately (`return "succeeded"`) if ANY recorded disposition is `succeeded`, before spawning anything. Only branches with `not_attempted` or absent keys are actually run (still through the same bounded-concurrency `worker()` loop).
9. After each `runOne` resolves, emit a checkpoint via a new engine-internal hook (thread `opts.checkpoint` down into `runParallel`, called with `{...entrySnapshot, transient: true, branchOutcomes: {...branchOutcomes}}` — clone the map each time so downstream consumers never see a live-mutating reference). This means `run()` must pass `opts?.checkpoint` into `runParallel(fork, ctx, shared, opts?.checkpoint, resume?.branchOutcomes)` (or thread `opts` through) — do NOT call the fork/exit checkpoints from inside `run()`'s existing checkpoint calls with `branchOutcomes` attached; only `runParallel`'s internal per-branch emissions carry it, and they must be `transient: true`.
10. Add `transient?: boolean` and `branchOutcomes?: Record<string, {disposition: "succeeded"|"failed"|"not_attempted"; text?: string; at: number}>` to `EngineCheckpoint` in types.ts (transient lives on EngineCheckpoint since it's a per-emission flag, not persisted run state; WorkflowRunState — which extends EngineCheckpoint — inherits both fields structurally, but the manager listener strips `transient` before persisting-as-non-terminal, per concern 03).

**executor.ts**
11. `NodeExecutor.runBranch?(node, ctx, signal?, branchKey?)` — add the 4th param (types.ts). `SingleAgentExecutor.runBranch` forwards it to `this.opts.spawnBranch?.(node, task, signal, branchKey)`; add `branchKey?: string` to the `spawnBranch` option's signature in `SingleAgentExecutorOptions`.

**workflow-driver.ts**
12. `BranchSpec` gains `branchKey?: string`. The `spawnBranch:` closure passed into `SingleAgentExecutor` (in `WorkflowDriver.start()`) forwards `branchKey` into `this.opts.fleet!.runBranch({..., branchKey})`.
13. The driver's `escalate` closure in `execRun()` (currently just emits a text message) gains the checkpoint param and, in addition to the existing message emission, emits a new frame `{type: "workflow_terminal", reason, checkpoint}` so the manager (concern 03) can persist the terminal marker. Keep the existing text-delta message emission for backward-compat display.
14. `WorkflowDriver`'s own `runId` (built in `execRun`, format `${this.opts.id}:${Date.now().toString(36)}`) is NOT yet applied to branch agent ids in this concern — that happens in concern 02's `spawnFleetBranch`, which needs the runId. Expose it via the existing `runId` the driver already tracks: thread it into the `fleet.runBranch` call as part of `BranchSpec` (`runId: this.runId`) so `spawnFleetBranch` can compute `br-<hash8(runId+":"+key)>-<slug(nodeId,12)>` without re-deriving it. Add `runId?: string` to `BranchSpec`. A tiny `hash8`/`slug` helper (FNV-1a or similar 32-bit hash → 8 hex chars; slug = `nodeId.toLowerCase().replace(/[^a-z0-9]+/g,"-").slice(0,12)`) can live in workflow-driver.ts and be exported for squad-manager.ts's spawnFleetBranch to import (or duplicate the 6-line helper — prefer exporting to avoid drift).
