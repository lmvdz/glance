# never-lose-work

## Outcome
- A workflow run that dead-ends (poison-cap resume exhaustion, visit-cap overflow with no route, no-progress-route exhaustion, or running off the graph) is marked terminal, surfaces as a sticky error via the existing catastrophe channel, and is never boot-looped through adoption/reconnect again.
- An operator can fork any checkpointed step of a prior run (default latest) into a brand-new run id that starts from the same code tip, with retry budgets reset for the failed fix-up tier so the fork is actually a fresh shot — not a guaranteed repeat of the same failure.
- Parallel fan-out branches get deterministic, collision-free agent ids derived from (runId, nodeId, visit, branch index); a cold daemon restart resumes a parallel node by re-spawning only the branches that never produced a real outcome, using the same ids so worktrees are reused automatically.
- The webapp TaskDetail view gets a 'Fork from step N' control next to Restart, backed by a read-only checkpoint history endpoint, gated on a persisted forkAvailable flag so old/new daemon-webapp combinations degrade safely instead of hiding or 404ing.
