# Append-only checkpoint log + persisted terminal marker with adoption/resume exclusion
STATUS: closed
PRIORITY: p0
REPOS: omp-squad
COMPLEXITY: architectural
TOUCHES: src/workflow/checkpoint-log.ts, src/squad-manager.ts, src/workflow/types.ts, src/types.ts

## Goal
Write a new append-only per-run checkpoint log (mirroring receipts.ts's JSONL convention) fed from the existing manager checkpoint listener (excluding transient branch-done emissions); wire the driver's new workflow_terminal event into a persisted workflowState.terminal marker that routes through the existing markCatastrophe channel and is excluded from resumable/reconnectLive/makeDriver/dispatch; derive dto.forkAvailable from the marker so it survives restarts; wire remove(deleteWorktree:true) to delete the run's JSONL.

## Approach
**New file src/workflow/checkpoint-log.ts** (mirror receipts.ts's shape/conventions — append-only JSONL, `path.join(stateDir, "workflow-checkpoints", `${runId}.jsonl`)`):
1. `export interface CheckpointLogEntry { seq: number; at: number; currentNode: string; outcome?: "succeeded"|"failed"; headSha?: string; vars?: Record<string,string>; }` — note `vars` is truncated (step 5) and this entry type deliberately excludes `visits`/`resumeAttempts`/`preferredLabel` (internal routing state, not shown to the picker) though the underlying WorkflowRunState written by the manager carries them; the log stores the FULL WorkflowRunState-shaped JSON per line (superset of CheckpointLogEntry) and the read-side projects down to CheckpointLogEntry for the GET endpoint (concern 04).
2. `export function checkpointLogPath(stateDir: string, runId: string): string` = `path.join(stateDir, "workflow-checkpoints", `${runId}.jsonl`)`.
3. `export async function appendCheckpoint(stateDir: string, runId: string, state: WorkflowRunState & {headSha?: string}): Promise<void>` — maintains a module-level `Map<string, {seq: number, chain: Promise<void>}>` keyed by runId (mirrors squad-manager.ts's `writeChain` promise-chaining pattern at line ~3514, but per-runId instead of one global chain). On first append for a runId in this process (i.e. no entry in the map yet), `seq` is initialized by reading the existing file's line count via `fs.readFile(...).then(t => t.trim() ? t.trim().split("\n").length : 0)` (0 if the file doesn't exist — `ENOENT` caught). Each append: truncate `state.vars.lastOutput`/`state.vars.lastText` to 4KB (`.slice(0, 4096)` with a `…(truncated)` suffix if longer) BEFORE serializing, mkdir the parent dir recursively, then `fs.appendFile(path, JSON.stringify({...state, seq: nextSeq, at: Date.now()}) + "\n")`, chained after the runId's prior promise (same pattern as `persist()`'s `writeChain`), incrementing the in-memory seq counter only after a successful append.
4. `export async function readCheckpoints(stateDir: string, runId: string): Promise<(WorkflowRunState & {seq: number; at: number})[]>` — reads the file, splits on `\n`, filters empty lines, `JSON.parse`s each with a try/catch that SKIPS (does not throw on) an unparsable trailing line (torn-tail tolerance for a hard crash mid-append), sorts by `seq` ascending.
5. `export async function deleteCheckpointLog(stateDir: string, runId: string): Promise<void>` — best-effort `fs.rm(checkpointLogPath(...), {force: true})`.

**squad-manager.ts wiring**
6. In `wire(rec)`'s existing `a.on("checkpoint", (state: WorkflowRunState) => {...})` listener (current line ~2651): after the existing `rec.options.workflowState = state; rec.dto.workflowState = state; this.emitAgent(rec); void this.persist();` lines, add `if (!state.transient && state.runId) void appendCheckpoint(this.stateDir, state.runId, {...state, headSha: await this.captureHeadSha(rec.dto.worktree)}).catch((err) => this.log("warn", `checkpoint-log append failed for ${rec.dto.id}: ${err}`));` — note the listener callback isn't currently `async`; make it `async (state: WorkflowRunState) => {...}` (EventEmitter tolerates async listeners; rejected promises are swallowed by the `.catch` already, but wrap the whole body defensively since this is a hot path). Add a small `private async captureHeadSha(worktree: string): Promise<string | undefined>` using the existing git-runner pattern from worktree.ts (`Bun.spawn(["git",...GIT_HARDEN_ARGS,"rev-parse","HEAD"],{cwd:worktree,...})`), best-effort (return undefined on any error) — this is the "data only, no rewind behavior this slice" capture.
7. Handle the new `"workflow_terminal"` frame type in `onAgentEvent`'s switch (near the existing `case "workflow_done":` at line ~2744): `case "workflow_terminal": { const {reason, checkpoint} = frame as {reason: string; checkpoint: EngineCheckpoint}; const runId = rec.options.workflowState?.runId ?? rec.dto.id; const terminal = {reason, at: Date.now(), forkPoint: {runId, seq: /* last appended seq for this run, tracked in checkpoint-log's in-memory map, or re-derive by reading the log length */ }}; rec.options.workflowState = {...(rec.options.workflowState ?? checkpoint as WorkflowRunState), terminal}; rec.dto.workflowState = rec.options.workflowState; this.markCatastrophe(rec.dto.id, terminal.reason); break; }` — `forkPoint.seq` should reference the checkpoint entry the fork will restore from; the simplest correct value is the seq of the checkpoint being persisted in the SAME emission (i.e. read `checkpointLogPath`'s current line count right before or via a small `getLastSeq(stateDir, runId)` export from checkpoint-log.ts, called synchronously in this handler after the checkpoint listener's own append settles — accept a small race and document it, or explicitly `await` the checkpoint append promise before computing `forkPoint.seq` since both events fire from the same driver call in quick succession).
8. Add `terminal?: {reason: string; at: number; forkPoint: {runId: string; seq: number}; supersededBy?: string}` to `WorkflowRunState` in `src/workflow/types.ts`.
9. Add `forkAvailable?: boolean` to `AgentDTO` in `src/types.ts`; derive it wherever `rec.dto.workflowState` is set from a terminal-marked state: `rec.dto.forkAvailable = !!rec.options.workflowState?.terminal && !rec.options.workflowState.terminal.supersededBy;` — set this alongside the `rec.dto.workflowState = ...` assignment in step 7, and ALSO recompute it in `derive()`'s neighborhood or wherever `rec.dto.status = this.derive(rec)` runs, so it's consistently refreshed (simplest: compute it once in the `workflow_terminal` handler and again whenever workflowState is reloaded from persistence in `adoptOrphanedAgents`/`restoreFromSnapshot`/`reconnectLive`, so a fresh boot recomputes it from the persisted marker rather than relying on it having been set in-memory before the restart).
10. Exclusions: `resumable` (current line 789) → `const resumable = (p: PersistedAgent): boolean => p.kind === "workflow" && p.workflowState !== undefined && !p.workflowState.terminal;`. `reconnectLive` (line ~757-785) → wherever it decides to auto-resume a live-thread workflow, add `if (p.workflowState?.terminal) continue;` (skip auto-resume of a terminal-marked run; it stays in the roster showing its sticky error, but its inner thread — if somehow still alive — is left alone rather than re-driven). `makeDriver`'s workflow branch (line ~2192-2205) → when building `resumeState: p.workflowState`, pass `resumeState: p.workflowState?.terminal ? undefined : p.workflowState` (a terminal run gets a driver with no resume state, i.e. it will not restart its graph walk if somehow prompted again) — actually simpler and safer: `makeDriver` should just never be called with intent to resume a terminal run because `resumable`/`reconnectLive`/adoption already filter it out; add a defensive assertion-comment rather than new branching logic, UNLESS `restart()` (concern 02) can reach a terminal agent — if so, guard `restart()` too: refuse (throw/no-op with a log) restarting a terminal-marked workflow agent, directing the operator to fork instead.
11. `remove(id, deleteWorktree)` (current line ~2623-2632): when `deleteWorktree` is true and `rec.options.kind === "workflow"` and `rec.options.workflowState?.runId`, add `await deleteCheckpointLog(this.stateDir, rec.options.workflowState.runId).catch(() => {});` alongside the existing `removeWorktree` call.
