# createInternal spawn path, deterministic branch agent ids, and stop-and-reprompt cold-resume reconciliation
STATUS: closed
PRIORITY: p0
REPOS: omp-squad
COMPLEXITY: architectural
TOUCHES: src/squad-manager.ts, src/types.ts

## Goal
Give squad-manager a private createInternal spawn path with a deterministic, attacker-unreachable id parameter; rebuild spawnFleetBranch to use it with ids derived from (runId, nodeId, visitIndex, branchIndex); implement the stop-and-reprompt reconciliation protocol so a cold resume of a parallel node tears down live roster agents whose branch keys won't match the resume's plan and re-prompts the ones that will re-run in their reused worktrees; close the restart() cold:true gap and the applyCommand unknown-command hole.

## Approach
**Internal create path (squad-manager.ts, near `create()` at line ~1990-2186)**
1. Define a module-private type `InternalCreateOptions` (NOT exported, NOT added to `CreateAgentOptions` in types.ts) = `CreateAgentOptions & {explicitId: string}`. Add `private async createInternal(opts: InternalCreateOptions, actor: Actor): Promise<AgentDTO>`. Its body is `create()`'s body factored so the `const id = newAgentId(name);` line (current line 2044) becomes `const id = opts.explicitId ?? newAgentId(name);` gated so ONLY `createInternal`'s call path can supply `explicitId` ŌĆö concretely: keep `create(opts: CreateAgentOptions, actor)` as the public wire-reachable entry (unchanged signature, still called from `applyCommand`'s `cmd.type==="create"` and server.ts's `manager.create({...cmd.options, track:true}, actor)`), and have it delegate to a new private `private async createWithId(opts: CreateAgentOptions, explicitId: string | undefined, actor: Actor): Promise<AgentDTO>` that contains today's full body with `newAgentId(name)` replaced by `explicitId ?? newAgentId(name)`. `create()` calls `this.createWithId(opts, undefined, actor)`. `createInternal()` calls `this.createWithId(opts, opts.explicitId, actor)` after checking `if (this.agents.has(opts.explicitId)) throw new Error(...)` (the duplicate-id rejection ŌĆö do this check BEFORE `createWithId` runs any worktree/spawn side effects). This keeps `explicitId` off `CreateAgentOptions` entirely ŌĆö `server.ts`'s `{...cmd.options}` spread (line ~1358) can never inject it because the field doesn't exist on the type the wire deserializes into, and `createWithId`'s `explicitId` param is only ever supplied by trusted in-process callers.
2. `createInternal` is the sole caller for both `spawnFleetBranch` (deterministic branch ids) and `fork()` (concern 04's new run id) ŌĆö no other caller should be added.

**Deterministic branch ids (spawnFleetBranch, current line ~2216-2241)**
3. `spawnFleetBranch(repo, parentId, spec: BranchSpec)` now receives `spec.branchKey` and `spec.runId` (added in concern 01, step 14). Compute `const id = spec.runId && spec.branchKey ? \`br-${hash8(spec.runId + ":" + spec.branchKey)}-${slug(nodeIdFromSpec, 12)}\` : undefined` ŌĆö reuse or reimplement `hash8`/`slug` from workflow-driver.ts (prefer importing the exported helper from concern 01 to avoid two implementations drifting). `nodeIdFromSpec` is `spec.name` (workflow-driver.ts's `spawnBranch` closure already passes `node.id` as `spec.name` ŌĆö confirm at `src/workflow-driver.ts` around the `spawnBranch:` line in `start()`).
4. Replace the current `const dto = await this.create({repo, name: spec.name, ...})` call with `const dto = await this.createInternal({repo, name: spec.name, model: spec.model, approvalMode: spec.approvalMode, parentId, autoRoute: false, bypassCap: true, task: spec.task, explicitId: id ?? newAgentId(spec.name)}, LOCAL_ACTOR)` ŌĆö pass `task: spec.task` so the branch's prompt is persisted on the `PersistedAgent` (already a plain field flow-through in `create()`'s body; no new persistence code needed beyond passing it here) instead of only being sent via `runAgentTask`'s live `agent.prompt(task)` call. This is what makes a branch re-promptable after a restart in step 8 below.
5. If `this.agents.has(id)` when `id` is defined (a re-spawn of the exact same branch ŌĆö e.g. resume calling spawnFleetBranch again for a `not_attempted` key whose old dead record is still in the roster from a prior boot), that's an expected "reuse" case, NOT a collision to reject ŌĆö but `createInternal`'s duplicate check (step 1) would throw. Resolve by having the RECONCILIATION step (below) `remove()`/`stop()` any stale roster entry for a key BEFORE resume calls spawnFleetBranch again, so by the time `createInternal` runs, the id is free. Document this ordering requirement in a comment on `createInternal`.

**Stop-and-reprompt reconciliation**
6. This is invoked when a `WorkflowDriver`/`WorkflowEngine` resumes a run whose `currentNode` is a parallel fork node (i.e., `resumeState.currentNode` resolves to a `kind:"parallel"` node) ŌĆö the reconciliation must run BEFORE `engine.run()`'s `runParallel` (concern 01) is invoked, because it needs to stop conflicting live agents first. Implement as a manager-side pre-step: in `makeDriver`'s workflow branch (current line ~2192-2205) ŌĆö or more precisely, wherever the manager decides to call `agent.start()` on a resumed workflow driver ŌĆö add `await this.reconcileParallelResume(p)` before starting the driver when `p.workflowState?.branchOutcomes` is present and `p.workflowState.currentNode` is a parallel node in the parsed graph (parse via the same `parseWorkflow`/`resolveWorkflowPath` machinery `makeDriver` already uses for `p.workflow.path`).
7. `reconcileParallelResume(p: PersistedAgent)`: parse the workflow, find the fork node matching `p.workflowState!.currentNode`, enumerate all branch edges to compute the full set of expected `branchKey`s the same way `engine.ts`'s `runParallel` will (same formula: `${forkId}#${visits[forkId] ?? 0}:${branchIndex}`, using `p.workflowState!.visits[forkId]` ŌĆö 0 if absent). For each expected key, compute its deterministic agent id (`hash8`/`slug`, same as step 3) and look it up in `this.agents`. Partition into: (a) keys whose recorded disposition (from `p.workflowState!.branchOutcomes`) is `succeeded`/`failed` ŌĆö these will NOT be re-run; (b) keys with `not_attempted`/absent disposition ŌĆö these WILL be re-run.
8. For EVERY live roster agent whose id matches an expected key in EITHER partition (a) or (b), call `await rec.agent.stop()` and remove it from the roster the same way `remove()`'s teardown does (stop + `this.agents.delete(id)`), WITHOUT deleting its worktree (branch agents reuse their worktree via `addWorktree`'s registered-path idempotency, per worktree.ts lines 80-83). This is the "stop-and-reprompt" step: partition-(a) matches are stopped because the join will use the recorded outcome, not their live state; partition-(b) matches are stopped because they'll be re-spawned fresh via `spawnFleetBranch` moments later (avoiding two roster entries under one deterministic id, and cleaning up any first_success crash-window loser `reconnectLive` may have already reattached ŌĆö `reconnectLive` has no parentId filter per the verified 761-778 range).
9. Re-prompting: partition-(b) branches are NOT explicitly re-spawned by `reconcileParallelResume` itself ŌĆö that happens naturally when `engine.run()` resumes and reaches `runParallel` (concern 01, step 8), which calls `runOne` ŌåÆ `executor.runBranch` ŌåÆ `spawnBranch` ŌåÆ `spawnFleetBranch` ŌåÆ `createInternal` with the now-freed deterministic id, in the SAME reused worktree (addWorktree's registered-path reuse), and the branch's persisted `task` (step 4) is what gets sent via `agent.prompt(opts.task)` inside `create()`'s existing task-prompt flow (line ~2162-2168) ŌĆö no new prompt-injection code needed. Add a one-line addendum to the branch's re-prompt: append `\n\n(Resuming after a restart ŌĆö prior partial work may already exist in this worktree; continue from where it left off.)` to `spec.task` inside `spawnFleetBranch` ONLY when `this.agents.has(id)` was true moments before reconciliation stopped it (track this via a `Set<string>` of ids reconciliation just stopped, passed through or checked against a short-lived manager field).

**Guard holes (squad-manager.ts)**
10. `restart()` (current line ~2600-2621): `const fresh = this.makeDriver(rec.options);` ŌåÆ `const fresh = this.makeDriver(rec.options, rec.options.kind === "workflow");` (or more precisely, pass `cold: true` whenever `rec.options.kind === "workflow"` ŌĆö `restart()` always rebuilds a fresh inner thread, so it must always resume cold for workflow agents to keep the poison cap active).
11. `applyCommand`'s `switch (cmd.type)` (current line ~2461 onward): add a `default: throw new Error(\`unknown command type: ${(cmd as {type?: string}).type}\`);` case so an old daemon receiving a new command type (e.g. `fork`, added in concern 04) errors loudly instead of silently no-oping (today the switch falls through with no default).
