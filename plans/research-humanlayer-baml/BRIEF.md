**Design tensions worth carrying forward**
- **Durable vs attached HITL.** HumanLayer built day-long durable approval threads because outer-loop agents detach; omp-squad's agents are attached children, so its in-memory pending is fine *until* you want off-dashboard, agent-initiated escalation (then F7's durable-thread shape starts to matter).
- **Step-count vs blast-radius safety.** F10 bounds risk by tiny agents; omp-squad bounds it by worktree+review. Don't conflate.
- **BAML's heavyweight core is the wrong shape here.** Rust+WASM+codegen buys cross-language identical behavior and an IDE playground — both irrelevant to one Bun/TS daemon. The crown jewel (SAP) is a ~100-LOC discipline when you only have a handful of small schemas.
- **HumanLayer deprecated the standalone approval SDK** ("a feature, not a product"). omp-squad already treats HITL as a feature of the control plane — the right call, validated.

## Part B — ABSTRACT (ranked transferable patterns)

### 1. One typed-decision helper with schema-aware coercion and a deterministic fallback
**Sources:** both (BAML typed functions + Schema-Aligned Parsing; HumanLayer Factor 4 / Factor 1)
**Pattern:** Route every "ask the model for a structured decision" through a single helper that takes a prompt, a tiny target schema (bool | literal-union/enum | object of typed fields | string[]), and a caller-supplied deterministic fallback — and *guarantees* it returns a valid value of that schema or the fallback, never raw model text.
**Mechanism:** `decide<T>({ prompt, system?, schema, fallback })` → `ompOneShot` → `extractJsonObject` → coerce against `schema`: for an enum, case-insensitive exact then substring snap (today's `snapToOption`); for a bool, a truthy regex (today's `TRUTHY_RE`); for an object, per-field string/enum coercion (today's `parsePlanJson` + `asApproval`/`asThinking`); on any miss or non-zero exit/empty/timeout, return `fallback`. The enum case MUST never return an out-of-set value (already the invariant in `parseDecision`). No scoring engine — first-valid-match is sufficient at this schema size.
**Value for omp-squad:** Collapses five hand-rolled parsers into one tested seam; every future LLM-decision site inherits the SAP-ish coercion + guaranteed-fallback discipline for free, instead of copy-pasting a sixth slice. Directly serves the "LLM decides what, deterministic code controls how, always with a fallback" pattern that *both* briefs converge on — the strongest recurring signal in this research.
**Where it applies:** core in `src/omp-call.ts` (alongside `ompOneShot`/`extractJsonObject`); migrate `src/supervisor.ts` (`parseDecision`/`snapToOption`/`chooseFallback`, supervisor.ts:64-121, `decide` :200-215), `src/smart-spawn.ts` (`parsePlanJson`/`asApproval`/`asThinking`/`infer`, :81-131), `src/intake.ts` (`extractDecision`/`llmRoute`, :69-89), `src/land.ts` (`defaultReviewer` APPROVE/REJECT, :284-292).
**Already have / gap:** The *transport* (`ompOneShot`) and *raw JSON grab* (`extractJsonObject`) are already deduped. What is NOT deduped — and is copy-pasted in shape across 4-5 sites — is the schema-aware coercion + guaranteed deterministic fallback. That's the new, unifying piece.
**Build vs Buy:** **Borrow the pattern.** Adopting BAML (Rust toolchain + codegen + new DSL) into a single-language Bun/TS daemon is all cost, no benefit — its cross-language and playground value props don't apply, and there are only a handful of tiny schemas.
**Size estimate:** ~120-180 LOC for the helper + coercers; net *reduction* once the 4-5 call sites are migrated. Small.

### 2. Off-dashboard, agent-initiated human escalation (omni-channel)
**Sources:** HumanLayer (Factor 7 + Factor 11; the original `human_as_tool` / OmniChannel product)
**Pattern:** A blocked or uncertain agent should be able to *reach a human where they are* (webhook/Slack/email), and to *deliberately ask* for advice as a first-class request — not only sit silently in a dashboard queue waiting to be noticed.
**Mechanism:** (a) A `NotificationSink` interface fired when a `PendingRequest` enters the queue — especially high-stakes ones, or any request left after the auto-supervise budget is spent — with a single webhook implementation (Slack/email are webhook variants). (b) Optionally a new agent-emitted request kind ("ask the operator") distinct from a blocking gate, carrying an `urgency` field (HumanLayer's `request_human_input` shape). The existing `answer` command path closes the loop unchanged.
**Value for omp-squad:** Turns "supervise by exception" into "be summoned by exception." Today, close the dashboard and a blocked agent stalls with no signal; this is the single capability most directly addressed by HumanLayer's core thesis and the biggest unlock for the autonomous/outer-loop fleet story.
**Where it applies:** `src/server.ts` (notification builder already exists at :133-135 and web-push at :456-459 — generalize to a sink); fire on enqueue / budget-spent in `src/squad-manager.ts` (`maybeAutoSupervise` :1180-1206); extend `PendingRequest` in `src/types.ts:30-45` with `urgency`; reuse `ClientCommand{type:"answer"}` (types.ts:499).
**Already have / gap:** Have: in-dashboard Queue, web-push notifications, the answer command. Gap: any *push to an external channel*, and any *agent-initiated* (vs gate-triggered) escalation with urgency.
**Build vs Buy:** **Borrow the pattern.** The HumanLayer SDK/cloud is deprecated; a sink interface + one webhook poster is a few hundred lines and avoids a hosted dependency.
**Size estimate:** ~100-150 LOC for the sink + webhook impl; +~30 for the urgency field + agent-initiated kind. Small-medium.

### 3. Prompt-regression tests as in-repo fixtures
**Sources:** BAML (`test` blocks with `@@assert`)
**Pattern:** Treat each model-decision parser as code with a versioned fixture table — `(raw model output, request) → expected decision` — asserted against the *parser + fallback*, not a live model call.
**Mechanism:** A plain `test_*.ts` (or assert-based `demo()`) feeding recorded/synthetic model outputs (clean JSON, fenced JSON, prose-wrapped, junk, empty) through the unified helper from Pattern 1 and asserting the coerced decision and the fallback. Deterministic, offline, no network — exactly ponytail's "one runnable check."
**Value for omp-squad:** Locks the riskiest code in the system — the coercion that *auto-approves agents* (`chooseFallback`/`isRiskyRequest`), where a false negative "can wreck main/prod" per the code's own comment — against silent regression. Cheap insurance on the auto-supervise path.
**Where it applies:** the pure functions in `src/supervisor.ts` (`parseDecision`/`snapToOption`/`chooseFallback`), `src/smart-spawn.ts` (`parsePlanJson`), `src/intake.ts` (`extractDecision`); and the new Pattern-1 helper.
**Already have / gap:** These functions were explicitly written to "test standalone" (their docstrings say so) but have no tests today. Pure gap, zero refactor needed.
**Build vs Buy:** **Borrow the pattern** — a fixtures table, no framework. Adopting BAML's playground is irrelevant (no `.baml` files).
**Size estimate:** ~80-150 LOC of fixtures. Small; pairs with Pattern 1 (one place to test once unified).

### 4. Comment-driven review on first-class plan/research artifacts
**Sources:** HumanLayer (Tasks = sessions+artifacts; "Humans and Agents collaborating on design docs — catch mistakes before 2,000 lines of code")
**Pattern:** Let a human review and *comment on the plan/research artifact itself*, and feed those comments back into the next phase's context — so the high-leverage review is the document, not the eventual diff.
**Mechanism:** Attach a comment/feedback thread to a Feature's artifact docs; on the RPI human gate, instead of a bare approve/revise select, carry the operator's inline comments forward as context into the Implement phase (or back into Plan).
**Value for omp-squad:** omp-squad already surfaces plan dirs as Features and gates phases — the missing piece is the *feedback channel on the artifact*. This is where review effort is cheapest (catch a wrong plan before 2,000 LOC) and aligns with the RPI workflow already shipped.
**Where it applies:** `src/features.ts` (plan-dir→Feature derivation, :100-192; `parsePlanConcerns` :166-192), the human gate in `src/workflow-driver.ts` (`extension_ui_request{select}`, header :13-18), and the web `Changes`/Feature views.
**Already have / gap:** Have: plan dirs as artifacts tied to Features, a diff panel, an approve/revise gate. Gap: an inline comment thread on the artifact and a path that feeds comments into the agent's next phase.
**Build vs Buy:** **Borrow the pattern.** UI + a small data model; no dependency fits.
**Size estimate:** Medium (UI-heavy): ~200-400 LOC across server, manager, and web.

### 5. Risk-tiered (stakes-ladder) approval policy
**Sources:** HumanLayer (Low/Medium/High stakes ladder)
**Pattern:** Classify each gated action into a small ordered set of stakes tiers and route by tier — low auto-approves, medium gets a cheap second opinion, high always reaches a human — instead of a single binary "risky or not".
**Mechanism:** Replace the binary `isRiskyRequest` boolean with a `riskTier(req) → low|medium|high`: low → `chooseFallback` auto-answer; medium → route to the model-backed `decide()` (already built in `supervisor.ts`) for a nuanced call; high (matches the destructive `RISKY_RE`, or `source==="tool"`) → always human.
**Value for omp-squad:** Cuts false human-interrupts on medium-risk actions while keeping the truly destructive ones human-gated — better signal in the Queue, fewer needless stalls, without loosening the high-stakes guard.
**Where it applies:** `src/squad-manager.ts` (`isRiskyRequest`/`RISKY_RE` :149-150,1203-1206; `maybeAutoSupervise` :1180), wiring the medium tier to `decide` in `src/supervisor.ts:200`.
**Already have / gap:** Have: a binary risky/not gate + a budgeted auto-answer + a separate model-backed supervisor that's currently all-or-nothing. Gap: the *tier* concept and the medium→`decide` routing connecting the two existing mechanisms.
**Build vs Buy:** **Borrow the pattern.** ~60-100 LOC. ponytail caveat: the binary gate already works — build this only if false-interrupt or over-blunt-approval rates are a measured pain, not preemptively.
**Size estimate:** Small.

### 6. Bounded retry / second-model attempt before the deterministic fallback
**Sources:** BAML (declarative `retry_policy` + `provider fallback`)
**Pattern:** On a transient model failure, retry (briefly) or try a fallback model *before* collapsing to the blunt deterministic decision — so a one-off hiccup doesn't silently cost decision quality.
**Mechanism:** An optional `{ retries?, backoffMs?, fallbackModel? }` on `ompOneShot` (or wrapped in the Pattern-1 helper): one bounded retry with backoff, then optionally a second model, then the caller's deterministic fallback. Keep it declarative and tiny — not BAML's nestable policy DSL.
**Value for omp-squad:** Today a transient empty/non-zero `omp` exit drops straight to a heuristic (e.g. intake picks `plain`, supervisor picks `chooseFallback`). A single retry recovers most transient failures and keeps decisions model-quality.
**Where it applies:** `src/omp-call.ts` (`ompOneShot` :12-26), consumed by `decide`/`infer`/`ompClassify`.
**Already have / gap:** Have: single-shot + deterministic fallback. Gap: any retry or model-fallback layer.
**Build vs Buy:** **Borrow lightly.** ~30-50 LOC. Optional — the fallback already keeps the fleet moving; this buys quality, not liveness.
**Size estimate:** Small.

### 7. Frequent Intentional Compaction + explicit context-quality priority
**Sources:** HumanLayer / ACE (40-60% window utilization; Correctness > Completeness > Size > Trajectory)
**Pattern:** Drive workflows around context budget — distill searches/logs/diffs into structured markdown at each phase boundary and keep window utilization mid-range, ranked by a stated priority (correct info first, then complete, then small).
**Mechanism:** At phase/stage boundaries, compact the prior phase's output into a structured artifact (omp-squad already has the primitives: `summarize`, `buildDigest`, `fenceUntrusted`) and encode the priority order where summaries are selected/truncated.
**Value for omp-squad:** Cleaner per-phase contexts in the RPI workflow; less noise carried between Research→Plan→Implement.
**Where it applies:** `src/summarizer.ts`, `src/digest.ts`, `src/context/extract.ts`, and the stage transitions in `src/workflow-driver.ts` / `src/workflow/engine.ts`.
**Already have / gap:** Have: the extractive/compaction *primitives* — but they serve cold-start resume and a cross-run heat graph, not live per-agent window management; the priority order is encoded nowhere. **Honest scope caveat:** the live in-flight window belongs to the `omp` harness, so the realizable value in the *control-plane* layer is limited to phase-boundary compaction.
**Build vs Buy:** **Borrow the pattern**, reusing existing primitives. Medium and partly out-of-layer — lowest-confidence item here.
**Size estimate:** Medium, but mostly wiring existing code; question whether the squad layer is the right owner before building.

### 8. Validate the stateless-reducer / unified-event-log model — and correct the federation analogy
**Sources:** both (HumanLayer Factors 5/6/8/12 + CodeLayer daemon topology; BAML's "reuse git/fs, don't reinvent")
**Pattern:** A control plane whose state is a pure reduction over one append-only event log, with TUI/web as thin clients of a single stream, is the validated target architecture — not something to rebuild.
**Mechanism:** N/A — this is a *don't-build* finding. Confirm the existing model (`agent_start…agent_end` → derived status; `audit.ts`/`receipts.ts`/`land-ledger.ts` JSONL; one core, two thin surfaces) matches CodeLayer's `hld` and BAML's reuse-the-platform ethos.
**Value for omp-squad:** Confirms direction (saves a speculative rebuild) and flags the one place the source analogy *breaks*: HumanLayer's local↔cloud sync is same-operator cross-**device** mobility; omp-squad's `federation.*` is cross-**operator** team coordination. They are orthogonal — do not let the roadmap conflate them or build a cloud-daemon/phone tier on the strength of the HumanLayer comparison.
**Where it applies:** `src/audit.ts`, `src/receipts.ts`, `src/land-ledger.ts`, `src/squad-manager.ts` (event→status reduction); `src/federation.ts` / `src/federation-sync.ts` / `src/coordinator.ts` / `src/presence.ts` (cross-operator, NOT cross-device).
**Already have / gap:** Have it all; the only "gap" is conceptual clarity on federation's scope.
**Build vs Buy:** **Neither — validate and document.** ponytail: a cross-device cloud-sync tier is speculative; skip until a real mobility need appears.
**Size estimate:** Zero (documentation/decision).

## Part C — Strategic verdict & /plan handoff

The research is a **directional validation, not a pivot signal.** omp-squad is already CodeLayer-shaped at the architecture level — one daemon owning roster/status, TUI + web as thin clients of a single event stream, worktree-per-agent isolation, derived-from-stream status, append-only audit/receipt ledgers, and HITL approval modes baked into deterministic code. HumanLayer's own arc *confirms* omp-squad's bet: they deprecated the standalone approval SDK because "HITL is a feature, not a product," which is exactly how omp-squad already treats it. The transferable value is therefore concentrated in two places: (1) **consolidating discipline omp-squad already practices ad hoc** — the five hand-rolled LLM-decision parsers are each a slice of BAML's typed-function-with-SAP, begging to be unified and tested; and (2) **one genuine capability gap** — HITL is dashboard-pull-only, so the autonomous-fleet/outer-loop story (HumanLayer's core thesis, Factors 7/11) needs off-dashboard, agent-initiated escalation. BAML should be **borrowed, never adopted**: a Rust+codegen DSL into a single-language Bun/TS daemon is pure cost, and its crown jewel (Schema-Aligned Parsing) is a ~100-LOC discipline at omp-squad's handful-of-tiny-schemas scale.
