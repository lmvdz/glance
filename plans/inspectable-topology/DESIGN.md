# Design: inspectable-topology

_Origin: /research of Apache Burr → plans/research-burr/BRIEF.md (patterns 6, 7). Adversarial design 2026-07-03: sonnet designer → 2× fable red team → fable arbiter._

## Approach
Keep the draft's C-shaped architecture — durable structural topology rides the roster snapshot (`PersistedAgent`/`AgentDTO`), while the existing sampled span/trace pipeline stays a detail layer — but rebuild the durability story, which both red teams correctly proved was asserted rather than designed. The verified reality is that every boot path reconstructs agents from explicit field lists (`create()` at src/squad-manager.ts:2091-2144 builds both `persisted` and `dto` as literals; `loadPersisted` ~3545+ and `adoptOrphanedAgents` ~808 re-enter `create()`; `attachExisting` ~859 hand-builds the DTO), so the slice's first and gating concern is boot-path threading: carry `parentId`/`parentNodeId`/`branchIndex`/`subagents`/`workflowGraph` through `CreateOptions`, both restore call sites, and the `attachExisting` DTO build, with a round-trip acceptance test per boot path (reconnect / adopt / loadPersisted): persist agent with all fields → boot via that path → assert fields on both DTO and next persisted snapshot. The same foundation slice adds chain-dedup inside `persist()` (if a write is already queued behind the in-flight one, return that queued write's promise — `persistNow` snapshots at write time so every awaiter still resolves after a write containing its state, preserving the `stop()` durability barrier at :739) replacing the draft's trailing-timer coalesce, plus a rate-limited warn log and a save-failure counter around `FileStore.save`'s currently-silent `try{}catch{}` (src/dal/store.ts:131-142), since the headline guarantee now rests on that call.

Lineage schema, corrected: roster-side lineage is `parentId` (exists) plus two new structured fields threaded through `BranchSpec` → `spawnFleetBranch` → `create()`: `parentNodeId` (the node in the parent's workflow graph this branch executes — renamed and re-documented per RT2, kept despite duplicating today's `spec.name` because name is a mutable display string and lineage must be structural) and `branchIndex` (distinguishes same-node siblings and cold-resume re-spawns). Subagent-side lineage is `subagents?: SubagentNode[]` (task/description truncated to 240 chars + `redact()`, mirroring the spans discipline; persist the spawn `index` so ordering and tracker reseeding survive), with three corrections that make it actually restart-surviving: (1) the dirty bit is set on any node creation or status *transition* inside `SubagentTracker.upsert`, not on lifecycle-typed frames — this closes the progress-carries-terminal-status race while still ignoring heartbeats; (2) flush is a merge-by-id (persisted ∪ live tracker, live wins per id) via one shared `mergeSubagents()` function, never an overwrite, and `attachExisting`/adopt seed the fresh tracker from `p.subagents` by finally wiring the dead `applySnapshot` (src/subagents.ts:94, zero callers today); (3) run-end closure — `finalizeRun`/abort/`agent_end` stamps non-terminal nodes `aborted` and flushes once, and `restart()`'s `subs.clear()` site does the same before clearing, so no persisted entry can claim `running` under a stopped agent forever. The read contract is single-sourced: `manager.subagents()` and the existing `GET /api/agents/:id/subagents` return the same merged projection the flush persists, eliminating the two-freshness-of-truth problem.

Workflow graph: new `workflow.graph` journal event carrying `WorkflowGraphSnapshot` (version:1; nodes include `kind/label/maxVisits/overflow/goalGate` **and `retryTarget`**, rendered as a dashed failure edge; edges with labels/conditions; start/exit/maxNodeVisits), emitted at the top of `execRun` immediately after `this.runId` is assigned (src/workflow-driver.ts:217) — not in `start()` — so it gets a real runId, fires once per run including resumes and second runs on the same driver, and never journals for a created-but-never-prompted agent. The consumer is a new `case "workflow_journal"` in `onAgentEvent` that writes `rec.dto.workflowGraph`/`rec.options.workflowGraph` and **returns** (matching the `subagent_` branch, avoiding the generic tail's derive/emit churn); all other journal event types remain deliberately unconsumed — general journal persistence is the brief's separate hooks-convergence initiative. Webapp work is a port, not greenfield: `webapp/src/lib/dto.ts` (which today lacks even `parentId`/`kind`/`workflowState`) gains the mirror fields; the lineage forest logic ports from the legacy `src/web/index.html` (:1260, :1323-1325) tree; dangling `parentId` promotes to root with an "orphaned (parent removed)" badge and `remove()` warn-logs when killing a parent with live children; the graph overlay (`buildWorkflowFlow` reusing PlanFlowDiagram's layout) merges `workflowGraph` + `workflowState` at render time. The trace drill-in stays but is redesigned around receipt *rollups* (never sampled, always present) with span waterfalls as opportunistic enrichment labeled as sampled, behind a server-side per-runId cache for finalized traces so the O(all-receipts) `readAllReceipts` scan isn't wired raw to a click path. Slice order: (1) boot-path threading + persist chain-dedup + save-failure visibility [gates everything]; (2) subagent merge-flush + lineage fields; (3) workflow.graph event + switch case; (4) webapp dto mirror + lineage tree + overlay + rollup-first drill-in [drill-in is the designated cut if slice 4 runs long]. Restart semantics are scoped honestly: the roster tree is *live* topology plus each agent's merged subagent history; dead pre-crash branch children (excluded from adoption by design, src/spawn-identity.ts:28-31) are historical and surface through receipts in the drill-in, not resurrected in the roster.

## Key Decisions
| Decision | Choice | Alternatives considered | Rationale |
|---|---|---|---|
| Where structural topology persists | Roster snapshot (PersistedAgent/AgentDTO via existing persist()/store.save), with explicit field threading through all three boot paths (create-opts passthrough for loadPersisted/adopt, DTO-build copy in attachExisting) and per-path round-trip tests | (a) Extend the span/trace pipeline to carry topology; (b) new sidecar topology.json; (c) the draft's version of this choice, which assumed fields 'ride along automatically' | Spans are 10%-sampled and only durable at finalizeRun — structurally wrong for an always-present fact. The roster snapshot is the right substrate but the draft's automatic-survival claim is verifiably false (create() builds from explicit literals; every prior additive field like scopeSource/workflowState was hand-threaded), so threading + tests become the slice's first, gating concern. Sidecar deferred as a measured contingency, not a default. |
| Write-amplification mitigation | Chain-dedup inside persist(): a call arriving while a write is queued behind the in-flight one returns that queued write's promise (max 2 writes per burst), no timer | Draft's 200-500ms trailing timer on topology-only persists; general debounce of persist(); per-agent diff writes | RT2's dedup is strictly better than RT1's timer-with-stop()-cleanup: zero new crash window, preserves the await-persist durability barrier stop() depends on (persistNow snapshots at write time), fixes the pre-existing per-checkpoint chattiness for free, and avoids the post-stop() cross-process last-writer-wins race RT1 identified — the exact OMPSQ-373..379 ordering-race shape. Where the red teamers disagreed, the evidence favors RT2. |
| Subagent dirty/flush semantics | Dirty on node creation or status transition in upsert() regardless of frame type; flush = merge-by-id (persisted ∪ live, live wins); tracker reseeded from p.subagents via the currently-dead applySnapshot on reattach/adopt; run-end/restart closure stamps non-terminal nodes aborted and flushes | Draft's lifecycle-frame-only dirty bit with unconditional overwrite flush; flushing on every changed frame; no closure | Verified in src/subagents.ts: LIFECYCLE_STATUS deliberately maps lifecycle into the progress vocabulary, so a progress frame can carry the terminal status first and make the lifecycle ingest a no-change — the draft's gate would lose terminal states permanently. Status transitions are rare (volume is heartbeats), so transition-dirty keeps write volume identical to intent. Merge+reseed+closure are what turn 'persisted once' into 'restart-surviving': without them the first post-restart event clobbers history and crashed children read running forever. |
| One read contract for subagents | Single mergeSubagents() projection: dto.subagents is its persisted snapshot, manager.subagents()/GET /api/agents/:id/subagents return the same merged live view, webapp reads the DTO from agent events | Two surfaces with different freshness (draft, implicitly); deprecating the endpoint; endpoint-only with no persisted copy | The existing live endpoint (src/server.ts:1218, polled by legacy UI) plus a lifecycle-boundary DTO copy is exactly the parallel-source-of-truth bug shape this codebase keeps paying for (~15 status-write bypasses of derive()). One merge function, used by both writer and reader, is the cheapest way to make them provably agree. |
| workflow.graph emission point and payload | Emit at top of execRun after runId assignment; snapshot includes retryTarget and version:1; consumed by a new workflow_journal switch case that returns (no fall-through to the generic tail); other journal types stay unconsumed | Draft's start()-time emission; consuming all journal event types now; a general journal log store | Both red teams independently converged: start()-time emission stamps the anchor event with the bogus `<id>:pending` runId and misses second runs on a reused driver (runId is only set in execRun, verified src/workflow-driver.ts:217). retryTarget is a real routing edge whose omission makes runs visibly 'jump' with no drawn edge. General journal persistence stays out — it is the brief's separate hooks-convergence initiative with its own ordering-design requirements. |
| Branch lineage fields | parentNodeId (renamed from parentStep, documented as 'the node in the parent's graph this branch executes') + branchIndex, threaded BranchSpec → spawnFleetBranch → create() | Draft's parentStep with wrong doc semantics; dropping the field as redundant with dto.name; deriving from name at render time | RT2 proved the draft's doc comment described the wrong thing and the value duplicates spec.name today — but name is a mutable display string (and identical across all siblings of a parallel node, per RT1), so structural lineage still needs its own field plus branchIndex to distinguish siblings and cold-resume re-spawns. Rename + redocument resolves the semantic bug; keep resolves the fragility. |
| Restart semantics of the tree (dead branch children) | Roster tree = live topology + per-agent merged subagent history; pre-crash branch children (excluded from adoption by design) are history, served via receipts in the trace drill-in; dangling parentId promotes to root with an 'orphaned' badge; no tombstone store | Per-run lineage tombstones outside the roster snapshot; adopting branch children (explicitly rejected by spawn-identity.ts to prevent double-lands); claiming full historical survival | The adoption exclusion is a correctness guard (double-land prevention), not an oversight — fighting it would reintroduce a landed bug class. buildTrace already reconstructs dead branches from finalized receipts' parentId, so history has a home; the design just stops overclaiming. Tombstones are real scope for a follow-on if operators need pre-crash topology without opening the drill-in. |
| Trace drill-in scope | Keep in slice 4 as the designated cut: rollup-first (receipt rollups are never sampled), span waterfall labeled as sampled/partial, server-side per-runId cache for finalized traces in front of readAllReceipts | RT2's cut-it-entirely; raising the workflow-run sample ratio; shipping the raw O(all-history) scan on a click path | The endpoint exists, works, and rollups (cost/duration/tool counts) survive sampling — that is most of what an operator drilling into a node wants. Cutting entirely leaves /api/trace dead code again; shipping raw wires an unbounded JSONL scan to a click. Rollup-first + cache + honest partial labeling threads it; sample-ratio changes are a telemetry-policy decision that does not belong inside a topology slice. |

## Risks
- Boot-path threading touches create()'s opts surface, which every spawn path shares — a mistake there regresses normal spawning, not just restore; the per-path round-trip tests are the guard and must land in the same PR as the threading.
- mergeSubagents union never prunes: an agent that spawns hundreds of subagents over a long life accumulates them all in every snapshot write. Bounded in practice (fan-out is tens) but worth a soft cap (e.g. keep newest N terminal entries) if state.json measurement shows growth; measure state.json size on the live fleet during slice 1.
- persist() chain-dedup changes timing observable by anything racing awaited persists; existing tests around stop()/upgrade barriers must pass unmodified, and a dedicated unit test (N concurrent persist() calls → ≤2 store.save invocations, all awaiters resolve after a write containing their state) is required.
- Lifecycle frames emitted while the daemon is down (detached host outlives daemon listener) are still lost — run-end closure and reattach reseeding bound the damage (no permanent 'running' lies) but a subagent that started and finished entirely during daemon downtime never appears; acceptable, documented, fixable later by wiring a get_subagents refresh on reattach.
- workflowGraph rollout gap: workflow agents already mid-run at deploy get no graph until their next resume/adopt (execRun re-entry); self-healing, one-time, noted for the PR description.
- Cold-resume duplicate branch children (same parentNodeId, new ids) remain visible in the tree as duplicates distinguishable only by branchIndex/timestamps — a pre-existing engine behavior this slice renders visible rather than causes; making it visible is arguably the feature.
- DbStore mode multiplies per-flush cost (full delete+insert of three table families + transcripts.json rewrite); chain-dedup caps rate but db-mode fleets at scale may still need the diff-upsert upgrade store.ts already forecasts — out of scope, flagged.
