# Transition-event subscription — retire the private lastStatus diff

STATUS: done
PRIORITY: p1
REPOS: omp-squad
COMPLEXITY: mechanical
TOUCHES: src/server.ts (maybePushAlert :2656, maybePushAlertOrg :2711, private `lastStatus`/`pushSeeded`/`lastPush` fields :557-568, boot seeding in `start()` :1024-1038), src/types.ts (SquadEvent "transition" variant :1333, TransitionEntry shape :30-50 — read-only reference), src/squad-manager.ts (`transition()` :8053, `recordTransition()` :8110, `getAgent(id)` :1986, reattach transition call sites :1702/:1845/:1861 — read-only reference), tests/push-server.test.ts, tests/push-org.test.ts, tests/escalation.test.ts

## Goal

`maybePushAlert` (file mode) and `maybePushAlertOrg` (DB mode, server.ts:2711) each keep a PRIVATE `lastStatus: Map<agentId, AgentStatus>`, populated by diffing successive `{type:"agent"}` SquadEvents against their own last-seen value, purely to recover the "prev status" `escalationPayload`/`completionPayload` need. That map duplicates state `SquadManager.transition()` (squad-manager.ts:8053) already computes, durably records (`transitions.jsonl` via `recordTransition` :8110), and already broadcasts as `{type:"transition", entry:{from, to, ...}}` (types.ts:1333) — emitted synchronously INSIDE `recordTransition`, strictly before the corresponding `{type:"agent"}` event a caller's own later `emitAgent()` call produces for the same change. This concern reads `entry.from`/`entry.to` straight off the canonical transition event instead of re-deriving them from a second, parallel, drift-prone diff. Pure refactor: the same transitions must produce the same pushes before and after — including the one known behavioral trap below, which must be preserved deliberately, not accidentally fixed as a side effect.

## Approach

- **New event source.** In `maybePushAlert` (server.ts:2656), add an `e.type === "transition"` branch that reads `entry.from`/`entry.to`/`entry.agentId` directly — no map lookup for the status diff. The current DTO fields the payload builders also need (name, pending, error, id, `completionPushArmed`) come from `this.singleManager?.getAgent(entry.agentId)` (squad-manager.ts:1986, returns the live `rec.dto` reference). By the time a transition event fires, `rec.dto.status` has already been set to `entry.to` inside `transition()` itself (it assigns `rec.dto.status = to` before calling `recordTransition`), and any preceding `setPending`/latch-exposure logic for the same frame (e.g. the `agent_end` case's `completionPushArmed` assignment, squad-manager.ts:6243-6248, which runs before the `derive()`-based `transition()` call that follows it at :6301) has already landed on `rec.dto` too — so `getAgent(entry.agentId)` at transition-event time returns a DTO exactly as consistent as the later `agent` event's own snapshot would be.
- **Drop the `e.type === "agent"` branch's diffing role.** The `agent` event branch in `maybePushAlert` no longer needs to read OR write `lastStatus` for the escalation/completion decision (that now happens entirely off the `transition` branch). Keep the `roster`-branch handling (:2660-2661) as the `pushSeeded` seed point — unchanged — but drop `lastStatus` writes there too once nothing reads the map. Confirm nothing else in server.ts depends on `lastStatus` before deleting the field (grep the class body) — the `start()`-time direct seed (:1024-1038) also writes it and must be revisited in the same pass (see the boot-safety note below).
- **`maybePushAlertOrg`'s twin (:2711)** gets the identical treatment: its `state.lastStatus` map (the per-org struct at :568) is retired the same way, reading `entry.from`/`entry.to` off the org-scoped `{type:"transition"}` events the registry's `onEvent` fan-out already delivers per org.
- **Boot-safety: a known behavioral trap to preserve, not "fix."** A reattach transition (squad-manager.ts:1845, `this.transition(rec, this.derive(rec), "reattach")`) fires on EVERY daemon restart for every surviving agent, even when `from === to` (a same-state call records under a non-`"turn-progress"` reason per `transition()`'s own documented semantics) — and when derive() computes a genuinely DIFFERENT status than what was persisted (e.g. a crash mid-run reclassified as `input` on restart), `entry.from !== entry.to` is TRUE for a boot-time replay, not a live event. Today's private-diff design happens to suppress a push flood here only as a side effect of `start()`'s direct seed (server.ts:1024-1038: `this.lastStatus` is seeded from `this.singleManager.list()`, which reflects the state AFTER reattach already ran, so the diff for that agent's first-ever `agent` event afterward reads `prev === a.status` — no push). A transition-event read has no equivalent accidental suppression: `entry.from` is the real pre-reattach status, always defined, so a reattach-time status flip would push at every boot unless deliberately gated. Preserve the OLD outcome (no push flood on boot) by gating the `transition`-branch on `pushSeeded` ALONE, exactly as the `roster`/`agent` branches already do today, and treat `pushSeeded` as "has this process finished its boot-time hydration," set once at `start()` (unchanged) — never derived from a per-agent map. This is the one point in the refactor where "preserve behavior" requires an explicit decision rather than a mechanical rename; write the regression test below to pin it.
- **`lastPush` (the 3s debounce map) and `pushSeeded` are untouched** — both are keyed/gated identically under the new design; only the "what is `prev`" question moves from a private diff to the canonical event.

## Resolution (2026-07-16)

Implemented as specced, with one seam update the doc predates: by execution time `maybePushAlert` dispatched BOTH `escalationPayload` and `completionPayload` (C01's generalized completion push), so the whole dispatch was subscribed to the transition event — `entry.from` feeds both builders, `getAgent(entry.agentId)` supplies the DTO, and `maybePushCompletionDone` (sync-disarm discipline included) is unchanged apart from where `prev` comes from. `lastStatus` is deleted from both lanes; `entry.denied` is skipped outright (a denied attempt never changed dto.status); a missing `getAgent` result fails closed.

Boot-safety decisions, made deliberate per the trap note above:
- File mode: the `transition` branch is gated on `pushSeeded` alone, and the field's comment now states its real meaning ("boot hydration finished", set once at start()). In the production ordering the reattach transitions can't even reach the lane (reconnectLive awaits every attachExisting before the server exists; the subscription is made inside start(), after the flag flips) — the gate is the deliberate second layer.
- Org lane: the lazy seed became an explicit boundary — the first event for an org (any type) marks hydration and NEVER alerts, replacing the old accidental self-suppression (the retired snapshot read the post-change roster, so the seeding event's diff always read prev === status; a transition event carries the real pre-change `from` and needed the boundary made explicit).

Verified:
- tests/push-server.test.ts: all prior cases re-driven through the transition lane (synthetic-event tests now use REAL resident agents so getAgent has a record); the two real-flow tests (promote category flip, duration gate) pass UNCHANGED — the genuine machinery emits the transition event itself. New: the MANDATORY boot-safety regression (full manager restart + a reattach-shaped `from!==to` transition delivered before the seed boundary → ZERO pushes; then a genuine post-boot escalation → exactly one push), a diff-the-two-lanes test (old-lane payloads computed directly from the same pure builders vs. what the server actually dispatches off the transition event — byte-identical across all four payload classes + a calm negative), and a denied-entry test.
- tests/push-org.test.ts: rewritten to transition events + fake `getAgent`; the lazy-seed test now pins the org-lane boot-replay boundary explicitly (a `reason:"reattach"` `from!==to` first event never alerts).
- tests/escalation.test.ts + tests/push.test.ts: pass UNTOUCHED — caller-side refactor confirmed.
- `tsc --noEmit` green; targeted transition-seam suites (transition-history, agent-lifecycle, lifecycle-settle-gate, lifecycle-enforcement, completion-push-arm, restart-reattach) green.
- LIVE (scratch daemon, isolated state/HOME per the skill): (1) a console agent's real starting→error transition paged the subscribed local push sink through the NEW lane (one encrypted aes128gcm POST); (2) SIGKILL + restart on the same state dir fired ZERO pushes across the whole boot; (3) a genuine new escalation post-restart paged normally (second sink POST). Note: the in-flight agent errored rather than sat in "input" (scratch HOME has no harness auth), so the live restart exercised the persisted-error/adopt boot path; the exact derive-mismatch reattach replay is pinned by the automated boot-safety regression instead.

Deliberate residual (documented in the org-lane comment): during a DB-mode tenant's multi-agent spin-up, transitions recorded after the org's seed boundary follow the canonical entry (push when `from !== to` into input/error) rather than the old per-agent map's replay-order-dependent diff. The old lane could both page spuriously and stay silent accidentally in that window; the invariant preserved is "the event that materializes the state never alerts", now stated in code rather than emergent.
