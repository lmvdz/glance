# Design: lifecycle-truth

_Origin: /research of Apache Burr → plans/research-burr/BRIEF.md (patterns 2, 3, 4). Adversarial design 2026-07-03: sonnet designer → 2× fable red team → fable arbiter._

## Approach
Ship "lifecycle-truth" as four concerns built on a guarded write-path with a reason-class transition model, replay-aware history, and crash-durable pending. Core: new pure module src/agent-lifecycle.ts exporting deriveStatus() (today's derive(), squad-manager.ts:3239) and canTransition(from, to, reason) built on a two-class reason model derived from the verified site table — Class D (derived: turn-progress, pending-add/answer/cancel derives) can never leave stopped/error, exactly preserving derive()'s stickiness and applyState's guard (3421); Class E (explicit: spawn, connect-begin/ok, restart, kill, abort, exit-clean, exit-error, fail, catastrophe, task/branch-start) is legal from any state including terminal→terminal, because the code verifiably does error→stopped on clean exit (2658-2659), any→error in markCatastrophe (1734) and fail (3247), and stopped/error→starting in ensureConnected (2375) — a flat RESET_REASONS whitelist is wrong. SquadManager.transition(rec, to, reason, cause?) and setPending(rec, next, reason, cause?, opts?) replace all 19 status + 5 pending writes via a per-site audit checklist (NOT a mechanical batch swap — both red teams proved that claim false): each site keeps its existing local guard verbatim (e.g. the exit handler's stopped-only check), transition() applies cause.error to dto.error itself before emitting (fixes the fail/markCatastrophe push-payload ordering), setPending takes an explicit target-status option for sites that manage status themselves (restart) and answerPending sets streaming=true before setPending so one input→working entry is recorded, not a phantom pair. Recording rules: same-state Class D calls early-return before ring/spool (hot-path property); same-state event-class calls (pending-*, catastrophe, fail) DO record, so a second question while already "input" and a repeat catastrophe enter the timeline; distinct-state denied attempts are spooled with denied:true plus a warn log — the guard is a bug detector, never a silent bug hider.

History persistence: a new small JsonlLog<T> module (hand-rolled ~60 lines on the automation-log ring+serialized-spool+hydrate idiom — do NOT refactor the live automation-log in this slice) writes stateDir/transitions.jsonl with size-capped rotation, torn-line-skipping hydrate, and a bounded in-memory ring; the consistency contract (ring authoritative for the tail, file best-effort) is documented in the module header. Every cause string and persisted pending title/message passes through redact() inside transition()/setPending()/persistNow() — the existing append() chokepoint (3428-3433) deliberately keeps secrets out of state at rest and the new surfaces must not reopen that hole. Timeline continuity: synthetic spawn/reattach/adopted entries at record construction, a best-effort daemon-stop entry in stop(), and adopted entries carry cause.priorId lineage (create() mints fresh ids on cold adopt, so the endpoint follows the lineage chain to stitch a crash-spanning timeline). Surfacing: transition() emits a dedicated low-frequency {type:"transition"} SquadEvent (which maybePushAlert can later subscribe to, replacing its private lastStatus diff — flagged follow-up, pushSeeded semantics preserved); AgentDTO.transitions is a tail capped at 5 significant entries (turn-progress excluded) so nothing meaningful rides the per-delta emitAgent hot path; GET /api/agents/:id/transitions serves the ring (file only on explicit request); a manager-maintained windowed rollup (e.g. errorTransitions1h on the DTO) feeds insights.ts, because a capped client-side ring provably undercounts busy agents.

Replay truth (the finding both red teams converged on and the draft missed): the agent-host replays up to 4000 ring frames on reconnect (agent-host.ts:15,188), which (a) already rebuilds pending with live, answerable correlation ids on warm reattach — so snapshot-based warm restore is CUT entirely — and (b) would pump phantom transitions into the history on every restart. Fix: a settle gate per reattached agent (seeded from the existing reattached set, 894; settle-point = agent.start() resolved plus one drain tick, verified by test) during which transition() records nothing to ring/spool, setPending skips persist, and maybeAutoSupervise is suppressed (also closing the pre-existing replay re-auto-answer hazard); after settling, one synthetic reattach entry summarizes the resulting state. Pendings added during the window are tagged replayed:true and stay answerable (the waiter lives in the surviving process), but two expiry rules resolve ghosts (replayed frames can resurrect already-answered questions, and a ghost permanently wedges applyState's reconciliation at 3419): (1) a live post-settle agent_end/turn boundary expires unanswered replayed pendings (a blocking ui request suspends the turn, so a completed live turn proves staleness); (2) the applyState poll expires replayed pendings after isStreaming===false holds for two consecutive polls. Expiry is never silent — it records pending-orphaned with the redacted question text in cause plus a transcript note. Durable pause (pattern 4): PersistedAgent gains pending (rides DbStore's JSON blob for free; FileStore needs no migration — ?? [] fallback), persistNow snapshots {...rec.options, pending: rec.dto.pending}, and setPending schedules a debounced persist (~1s coalesce; persistNow serializes all transcripts per call, verified 3538-3541, so per-mutation full persists are too heavy — the ≤1s crash window is documented best-effort and graceful stop() already awaits the chain as a durability barrier). Cold adopt consumes persisted pending only to record pending-orphaned closes (done inside adoptOrphanedAgents/loadPersisted after create() returns — CreateAgentOptions does NOT gain pending) with a reask-expected marker when workflowState guarantees the gate re-asks; the draft's restored:true-but-answerable machinery is deleted as internally contradictory. Scope cuts: workflow_journal interleaving (pattern 7) and the server.ts alert refactor are out, filed as follow-ups; enforcement is a bun test that parses squad-manager.ts for .dto.status=/.dto.pending= assignments outside the two methods and the whitelisted DTO-literal construction sites (a CI grep is bypassable and rtk mangles bash grep output locally). Concern order: (1) lifecycle module + guarded write-path + per-site swap + settle gate + enforcement test; (2) JsonlLog + transitions.jsonl + transition event + endpoint + redaction; (3) webapp timeline strip in TaskDetail.tsx + dto.ts mirror + insights rollup; (4) durable pending + cold-adopt orphan-close + ghost expiry.

## Key Decisions
| Decision | Choice | Alternatives considered | Rationale |
|---|---|---|---|
| Transition legality model | Two reason classes: derived reasons (turn-progress, pending-* derives) can never leave stopped/error; explicit reasons (spawn, connect-begin/ok, restart, kill, abort, exit-clean, exit-error, fail, catastrophe, task/branch-start) are legal from any state; call sites keep their existing local guards verbatim | Draft's flat terminal-stickiness with RESET_REASONS={restart,kill,spawn}; Burr-style declared edge graph with halt-on-no-match | Code-verified: four sites legally overwrite terminal states today (exit handler error→stopped at squad-manager.ts:2658, markCatastrophe any→error at 1734, fail any→error at 3247, abort any→stopped at 2256) and ensureConnected walks stopped/error→starting (2375) — the draft's table would silently swallow catastrophe summons and wedge reconnecting agents at 'stopped'. The class model reproduces today's behavior at all 19 sites exactly; retained site guards mean transition() is a floor, not a rewrite of per-site semantics. |
| Recording semantics for same-state and denied transitions | Same-state derived calls early-return before ring/spool; same-state event-class calls (pending-*, catastrophe, fail) record entries; distinct-state denied attempts spool with denied:true + warn log | Uniform from===to early-return (draft); Burr's loud halt on illegal; fully silent no-op on denied | The uniform early-return erases the slice's headline deliverable (second question while 'input', repeat catastrophes with new detail); spool growth comes solely from turn-progress, so exempting event-class reasons costs nothing. Silent denial hides bugs — recording denials turns the guard into a bug detector. |
| Warm-reattach pending restore | Cut snapshot-based warm restore entirely; agent-host ring replay is the source of truth (live correlation ids), pendings re-added during the settle window are tagged replayed:true and remain answerable; ghosts expire via (a) live post-settle turn boundary or (b) isStreaming===false across two polls, always recorded as pending-orphaned with the question text | RT1's reconcile-both-sources approach; draft's persisted-snapshot restore into dto.pending; do nothing (accept the pre-existing ghost wedge) | Verified: agent-host.ts:15,188 replays up to 4000 frames on connect and rpc-agent.ts:274 re-emits ui frames with no suppression — the operator already sees the question again with a live waiter, so snapshot restore duplicates it and adds stale entries. Immediate persist is too heavy (persistNow serializes every agent's full transcript, verified 3538-3541); a sidecar splits roster restore across two sources of truth. | Wire surfacing and insights feed | Dedicated {type:"transition"} SquadEvent from transition(); AgentDTO.transitions capped at 5 significant entries (turn-progress excluded); windowed error-transition rollup computed manager-side and shipped as a small numeric DTO field for insights.ts; full history via GET /api/agents/:id/transitions following priorId lineage | Draft's ~20-entry DTO tail riding every emitAgent; client-side errorTransitionCount over the DTO ring | emitAgent fires per RPC frame including message_update deltas (2762-2764) and each event is stringified per WS client — a 20-entry tail with cause strings is a bandwidth regression on the hottest path for data that changes almost never. | Redaction | redact() applied to every cause string (title/error/message) and to persisted PendingRequest text fields inside transition()/setPending()/persistNow() | Spool raw and redact at read/display time | append() (3428-3433) is explicitly the chokepoint keeping secrets out of state at rest; error strings embed command output and onHostTool stringifies raw tool args into message (3186). ## Risks
- The ghost-expiry poll rule assumes omp reports isStreaming===false only when NOT suspended on a blocking ui request; if omp reports non-streaming while blocked, a genuine question could be expired on reattach. Mitigation: expiry is never silent (pending-orphaned entry + transcript note with the question text), the turn-boundary rule is independent of this assumption, and concern 4's acceptance test must exercise a live omp blocked on a confirm across a daemon restart before the poll rule ships.
- The settle-point ('agent.start() resolved plus one drain tick') is a heuristic for when the host's ring replay has drained; if replay frames straggle past it, a few phantom transitions leak into history after restarts. Mitigation: a dedicated test drives attachExisting against a scripted host replaying a large ring and asserts zero recorded transitions during the window; the synthetic reattach entry marks the boundary so stragglers are at least visually attributable.
- Debounced persist leaves a ≤1s window where a crash loses the newest pending mutation — accepted as documented best-effort (strictly better than today's never-persisted), with graceful stop() still a hard barrier.
- The per-site audit (19 status + 5 pending sites) is the concern most exposed to subtle behavior drift; the plan must carry the verified site table as a checklist with expected reason + retained guard per site, and the enforcement test locks the door behind it.
- transitions.jsonl rotation discards old history by design; the full forensic record beyond the rotation cap is receipts/transcripts, not this log — stated in the module header so nobody builds long-horizon analytics on it.
- webapp/src/lib/dto.ts is a hand-maintained mirror already missing backend fields; the new transitions/rollup fields must be added there in concern 3 or the UI silently type-drops them.
