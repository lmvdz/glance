## 1. Scout brief — what TencentDB-Agent-Memory is

A **local-first, layered agent memory system** shipped as an npm plugin (`@tencentdb-agent-memory/memory-tencentdb`) for the OpenClaw agent runtime (+ a Python/Hermes HTTP-gateway adapter). Despite the name it is **not** a hosted TencentDB product — the default backend is fully local SQLite (`sqlite-vec` + FTS5) with a local GGUF embedding model (`embeddinggemma-300m`, 768-dim); a Tencent Cloud VectorDB (TCVDB) backend is optional.

Two independent subsystems:

### Long-term memory — L0→L3 semantic pyramid (`src/core/`)
- **L0 Conversation** — raw messages verbatim (SQLite / JSONL). Source of truth, never destructively summarized.
- **L1 Atom** — LLM extracts structured "memory records" every N turns (default 5) or after idle timeout. Each record: exactly one `type ∈ {persona | episodic | instruction}`, numeric `priority` 0-100 (−1 = strict global instruction), a `scene_name`, `source_message_ids` (traceability), free-form `metadata` (episodic carries `activity_start/end_time`). Dual-written to append-only JSONL (source of truth) + vector store.
- **L2 Scenario** — a **tool-enabled LLM agent**, sandboxed to a `scene_blocks/` dir, reads/writes human-readable Markdown "scene block" files aggregating related L1 atoms into narrative scenes. Capped at `maxScenes=15`: **at the cap the LLM is forced to merge the most-similar/least-active scenes before creating new ones** — this *is* the compaction/forgetting mechanism.
- **L3 Persona** — a single `persona.md`, regenerated by LLM every 50 new memories, **incrementally** (only scenes updated since last generation are fed in). Backed up before each rewrite.
- **End-to-end traceability**: Persona → Scene (`result_ref`) → Atom (`source_message_ids`) → raw L0. "Compressible but auditable back to raw evidence." Lower layers always kept as ground truth.

### Short-term / in-session symbolic memory — "context offload" (`src/offload/`)
- During long tool-loops, verbose tool outputs are offloaded to `refs/*.md` and **replaced in-context by a compact Mermaid diagram** ("MMD") of task-state transitions, each node carrying a `node_id`. The agent reasons over the small graph and re-reads a ref file (by `node_id`) only to verify a raw detail.
- Triggered at context-window ratios: mild compression at 0.5, aggressive at 0.85; injected MMD bounded to 0.2 of context budget. MMD inserted as a marked message **at a point that never splits a `tool_call`/`tool_result` pair**. Benchmarked: 30-61% token reduction, +8-52% relative pass-rate (WideSearch/SWE-bench/AA-LCR).
- **Critical caveat for omp**: this operates *inside the agent's own live context window* (it's a plugin running in the agent runtime).

### Retrieval, dedup, safety
- Retrieval: `keyword` (FTS5 BM25) | `embedding` (cosine) | `hybrid` (both merged via Reciprocal Rank Fusion, k=60). Score threshold 0.3 with a small-corpus fallback that trusts FTS5 ranking.
- **Write-time LLM dedup/merge**: before storing a new atom, recall a candidate pool and one batch LLM call decides per-memory `store | update | merge | skip`. Merges mutate the derived index in place; JSONL stays append-only.
- **Prompt-cache-aware context splitting**: stable content (persona + scene-nav + tools guide) → cacheable system prefix; fresh recalled memories → per-turn prepend, so recall never invalidates the system-prompt cache.
- **Fail-soft**: store methods return empty/false rather than throwing; recall has a 5000ms hard timeout that *skips* injection rather than blocking the turn.
- **Retention safety guards**: time-based deletion with minimum-retained floors (L0≥50, L1≥20) and a circuit-breaker that blocks deletion if `expired/total > 80%` (added after a real data-loss incident).

**Hard architectural constraint (documented, from the Mastra Code research / PR #69):** the daemon **cannot touch a child agent's live context window** — any memory layer it adds is a **turn-boundary / cross-process** layer, never in-window compaction.

## 3. Comparator table (concept → transferability)

See conversation. Key rows: L0→L3 pyramid (transferable *reshaped* — omp has no single evolving subject; the analog is per-repo institutional memory), typed traceable atoms, forced-merge-at-cap compaction, incremental delta-fed regeneration, Mermaid in-window offload (blocked by the daemon constraint above), hybrid+RRF (needs vector infra omp lacks), write-time dedup/merge (valuable *because* concurrent writers), prompt-cache splitting, fail-soft timeout, retention guards.

**Six tensions**: (1) LLM-in-write-path cost × fleet parallelism → pin cheap tier + batch/async; (2) single persona vs multi-tenant ephemeral fleet → per-repo conventions is the real analog; (3) sequential single-writer vs concurrent multi-agent writers → needs race handling; (4) BM25-only vs vector infra → fusion logic cheap, infra separate & unproven ROI; (5) ephemeral worktrees vs durable curator storage; (6) omp sits at the **opposite** failure pole (unbounded growth, not over-deletion).

## 4. Strategist — ranked patterns (opus; fable was rate-limited)

### Rank 1 — Fleet institutional-memory consolidation (async dedup/merge of decisions & facts)
**Pattern**: When many independent writers append durable facts about the same subject, a periodic background pass recalls near-duplicate/contradicting records and consolidates them (`keep | update | merge | drop`) using a cheap model, **out of the write path** — so the derived knowledge base stays crisp instead of accreting redundant, conflicting entries.
**Mechanism**: A daemon-level loop (not a per-write gate) periodically groups a repo's `decisions`/fact records by lexical similarity (BM25 candidate pool over the existing fabric docs — no embeddings needed), sends each cluster to a cheap one-shot LLM (haiku tier, the `reflection.ts` template) that returns a consolidated fact + the `source` ids it subsumes, and rewrites the derived store while the append-only JSONL stays ground truth.
**Value for omp-squad**: The cold-start **primer is consumed by every spawned agent** (`squad-manager.ts:3040`), and its `decisions` facts are appended per-feature with **zero dedup** (`fabric.ts:276`, `f.decisions ?? []`). Under fleet parallelism this is omp's *genuine, source-tool-never-faced* failure mode: 5 agents each recording "we chose X" pollute every future agent's starting context. Fixing it has fleet-wide leverage — it improves the input to every cold start.
**Where it applies**: `src/fabric.ts` (decisions assembly), the `PersistedFeature.decisions` write sites, a new daemon background pass alongside `src/worktree-reaper.ts`/scout loops, cheap-LLM plumbing reused from `src/reflection.ts`.
**Build vs Buy**: **Borrow the pattern.** No dependency — reuse existing BM25 (`fabric-search.ts`) for candidate recall and the existing haiku path.
**Effort/Risk**: Medium. Main risk = LLM merge *loses* a real distinction (mitigate: JSONL stays authoritative + traceability from Rank 2; consolidation is reversible).

### Rank 2 — Typed, traceable institutional memory + incremental per-repo conventions doc (the reshaped L3)
**Pattern**: Give durable memory records an explicit **type** and **priority**, and a backlink to the raw evidence that produced them; periodically distill them into a single human-readable "conventions/identity" document that is **regenerated incrementally from only what changed** and **backed up before each rewrite**.
**Mechanism**: Extend fact/decision records with `type ∈ {decision | convention | instruction | episodic}` + `priority` + `sourceReceiptId`/`sourceTurnId`. A low-frequency pass (every N new facts) feeds *only records changed since the last run* to a cheap LLM that rewrites `conventions.md` per repo, backing up the prior version first. Traceability chain: `conventions.md` entry → fact record → receipt line → transcript turn.
**Value for omp-squad**: omp's "subject" is not a person — it's a **repo's accumulated institutional knowledge**. A per-repo `conventions.md` gives every cold-started agent the repo's hard-won norms at zero turn cost, strengthening the existing primer. Typing + backlinks also feed omp's existing **UI trust-legibility** work (why did the fleet decide X → drill to the receipt/turn). Incremental-delta-feed + backup-before-rewrite keep the LLM cost bounded and the doc safe.
**Where it applies**: `src/fabric.ts` (`FabricDecisionFact` gains type/priority/source), `src/receipts.ts` (backlink target), `src/digest.ts` (digest entries backlink to receipt lines), `src/fabric-search.ts` (type filter already exists — `KbDocType`), webapp Knowledge panel (`webapp/src/components/KnowledgePanel.tsx`) surfaces type/provenance.
**Build vs Buy**: **Borrow.** Builds directly on Rank 1's consolidation pass.
**Effort/Risk**: Medium-high (schema + incremental regen + backups). Risk = scope creep toward a full pyramid; keep it to one flat `conventions.md` per repo, not L2 scene files.

### Rank 3 — Retention with safety guards for receipts & digests
**Pattern**: Bound append-only stores with time-based deletion **gated by floors and a circuit-breaker** — never delete below a minimum retained count, and refuse to delete if the expired fraction exceeds a ceiling (a runaway-clock guard).
**Mechanism**: A retention pass over `receipts/<agentId>.jsonl` and `digests/*.md`: keep the last M receipts or T days, whichever retains more; **minimum floor** (e.g. ≥50 receipts/agent); **block the whole delete if `expired/total > 80%`** (guards against a bad clock/config nuking everything).
**Value for omp-squad**: Closes a **code-acknowledged ceiling** — `receipts.ts:194` explicitly flags "no rotation/retention." omp is at the *opposite* failure pole from the source tool's over-deletion incident, so importing their **guard-rail shape** lets omp add retention without swinging past it.
**Where it applies**: `src/receipts.ts` (new prune fn), `src/digest.ts`, wired into the existing retention cadence near `src/ttl-registry.ts`/`worktree-reaper.ts`.
**Build vs Buy**: **Borrow the guard-rail shape.** Pure local code.
**Effort/Risk**: Low-medium. Low risk given the floors + circuit-breaker are the whole point.

### Rank 4 — Offload-don't-destroy for the daemon transcript ring
**Pattern**: When a bounded in-memory buffer must evict, **spill evicted entries to disk** (retrievable by id) instead of dropping them, so the long-horizon record stays complete and auditable.
**Mechanism**: When `rec.transcript` exceeds `MAX_TRANSCRIPT=800` (`squad-manager.ts:5839`), append the evicted head to a per-agent `transcript-archive/<agentId>.jsonl` before shifting it out; the digest/UI can page in older turns on demand.
**Value for omp-squad**: Today eviction is **lossy** — evicted turns vanish from `transcripts.json` and from any digest built afterward, so long-running agents lose their own history.
**Honest limitation**: This is the *weak* version of the source tool's Mermaid offload. The high-value in-window version (compact-graph-replaces-verbose-output) **does not port** — the daemon can't touch a child agent's live context window (the documented hard constraint). So the payoff here is *auditability/digest completeness*, **not** the benchmarked token/pass-rate win. Rank accordingly.
**Where it applies**: `src/squad-manager.ts` (`append`/ring cap), `src/dal/store.ts` (`saveTranscripts`), `src/digest.ts`.
**Build vs Buy**: **Borrow.** Local JSONL spill.
**Effort/Risk**: Low. Risk = disk growth → fold into Rank 3's retention.

### Rank 5 — Cheap hardening bundle
- **Fail-soft timeout on fabric assembly at spawn**: wrap `buildFabricSnapshot`/`buildContextPrimer` (`squad-manager.ts:3040`) in a short timeout that skips the primer rather than delaying spawn. Low effort, pure safety.
- **Idle-timeout digest trigger**: `digest.ts` currently builds only at `finalizeRun` (`squad-manager.ts:4865`); add an idle-timeout trigger so long-idle-but-unfinished agents get a resumable digest before a possible crash. Ties into the existing cold-adopt path.
- **Prompt-cache-aware split (design guidance, not a gap yet)**: omp injects the primer once at spawn via `appendSystemPrompt` (already the stable prefix). If/when per-turn recall is added, keep stable content in the system prefix and fresh recall in a per-turn prepend so recall never busts the prompt cache. Bake in from day one.

## 5. CUT (clever but doesn't fit omp-squad)

- **Hybrid retrieval + RRF + embeddings/vector store** — omp deliberately has no vector infra; ROI unproven on a code/identifier corpus where BM25 already suits exact-match. The *RRF fusion logic* is cheap but presupposes a second ranker that doesn't exist. **Buy call is too heavy for unproven gain.** (Even the "small-corpus fallback" is a non-gap — `fabric-search.ts` already avoids over-thresholding and marks weak matches.)
- **Mermaid in-window context offload (as a daemon feature)** — blocked by the documented hard constraint (daemon can't touch child context window). Only the weak "don't destroy on eviction" residue survives → Rank 4.
- **Personality-style L3 persona** — no single evolving subject in a fleet of ephemeral agents; reshaped into the per-repo `conventions.md` (Rank 2).
- **Heavyweight sandboxed curator *agent* role** — overkill; the consolidation is a daemon background pass (Rank 1), not a full standing agent with a worktree.

---

## 6. Recommended /plan slice (chain these first, in order)

1. **Rank 1 — async fleet dedup/merge of decisions/facts.** Highest leverage: improves the primer *every* cold-started agent consumes, and addresses omp's genuine fleet-specific failure mode (concurrent writers polluting institutional memory). Foundation for Rank 2.
2. **Rank 2 — typed + traceable facts + incremental `conventions.md`.** Builds on Rank 1's consolidation; turns the consolidated store into a durable, auditable per-repo knowledge doc that feeds the primer and the trust-legibility UI.
3. **Rank 3 + Rank 4 + Rank 5 as one "hardening + retention" bundle.** Closes the flagged unbounded-receipts ceiling with safety guards, stops lossy transcript eviction, and adds the cheap fail-soft/idle-trigger ports. Low-risk, mostly-local, ships alongside 1–2.

Ordering rationale: institutional-memory *quality* (1→2) is the fleet-wide-leverage win and the conceptual core of what the source tool got right, reshaped for a multi-tenant fleet; *durability/safety* (3→5) is real but lower-leverage and naturally bundles. The vector-store and in-window-offload temptations are explicitly cut so /plan doesn't chase infra with unproven ROI or fight a documented architectural constraint.
